# 印刷企业内部知识库问答系统：文档入库细节与质量把控

## 1. 这篇文档回答什么问题

这篇文档用于回答面试官关于“文档怎么入库”“OCR 结果怎么把控质量”“解析错误怎么处理”的追问。

核心口径是：

> 文档入库不是把原始 PDF 直接切块后丢进向量库，而是先经过格式识别、文档解析、OCR、质量评分、清洗去重、脱敏、切块、向量化和入库。  
> 对扫描件、复杂 PDF、表格资料，系统会用 PP-Structure 做版面分析和 OCR，并基于返回的置信度做质量分级，低质量内容不直接进入正式知识库。

## 2. 文档入库整体流程

文档入库属于离线流程，目标是把企业内部资料转换成可检索、可追溯、质量可控的知识片段。

```text
原始资料
  -> 文件类型识别
  -> 文档解析 / OCR
  -> 解析质量评分
  -> 清洗、去重、脱敏
  -> 元数据补充
  -> 父子块切分
  -> BGE-M3 向量化
  -> 写入 Milvus
  -> 进入 RAG 检索链路
```

常见资料类型包括：

| 资料类型 | 处理方式 | 入库目标 |
| --- | --- | --- |
| FAQ Excel / CSV | `pandas` / `openpyxl` 解析 | 写入 MySQL 标准问答库 |
| 可复制文字的普通 PDF | `pdfplumber` 提取文本 | 进入清洗和切块链路 |
| 扫描件 / 图片 PDF | PP-Structure / PaddleOCR OCR | 提取文字、版面和置信度 |
| 图文混排 / 表格资料 | PP-Structure 结构化解析 | 保留标题、正文、表格和版面结构 |
| Word / PPT | 对应 loader 提取文本，必要时 OCR | 统一转换为 Markdown / JSON |

面试时不要说“所有文档都用 OCR”。更准确的说法是：文本型资料优先用轻量解析，只有扫描件、图片类资料、复杂图文资料才进入 OCR 或结构化解析链路。

## 3. 为什么选择 PP-Structure 做主解析引擎

PP-Structure 是 PaddleOCR 提供的文档结构化分析能力，可以同时做版面分析、OCR 和表格识别。

选择它的主要原因是：

1. **有置信度**：每个文本块或表格单元格都有 `confidence`，方便做质量评分。
2. **有版面结构**：能区分标题、正文、表格、图片等区域。
3. **表格更好处理**：表格可以转成 HTML 或结构化单元格，减少行列关系丢失。
4. **适合质量把控**：可以从返回结果中统计平均置信度、低置信比例、结构完整性等指标。

对比纯 PaddleOCR：

| 维度 | 纯 PaddleOCR | PP-Structure |
| --- | --- | --- |
| 核心能力 | 文字识别 | 版面分析 + OCR + 表格识别 |
| 返回结构 | 文本框坐标、文字、置信度 | 类型、坐标、OCR 结果、表格结构、置信度 |
| 是否知道标题/正文/表格 | 不知道 | 知道 |
| 表格处理 | 只能识别文字和坐标 | 可返回表格 HTML 和单元格信息 |
| 质量评分 | 可以基于文字置信度评分 | 可以结合置信度和结构完整性评分 |

## 4. PaddleOCR 和 PP-Structure 的返回格式

### 4.1 纯 PaddleOCR 返回什么

纯 PaddleOCR 更像“识字工具”，返回的是每个文本框的位置、识别文字和置信度。

示例结构：

```python
[
    [
        [
            [[24.0, 36.0], [304.0, 34.0], [304.0, 72.0], [24.0, 74.0]],
            ("印刷企业管理规范", 0.9823)
        ],
        [
            [[28.0, 90.0], [210.0, 88.0], [210.0, 116.0], [28.0, 118.0]],
            ("第一章 总则", 0.9654)
        ]
    ]
]
```

可以理解为：

```text
result = [
  [
    [bbox, (text, confidence)],
    [bbox, (text, confidence)]
  ]
]
```

字段含义：

| 字段 | 含义 |
| --- | --- |
| `bbox` | 文本框四个角坐标 |
| `text` | OCR 识别出的文字 |
| `confidence` | 当前文本框识别置信度，通常是 0 到 1 |

纯 PaddleOCR 的问题是：它知道“这里有一行文字”，但不知道这行文字是标题、正文还是表格里的单元格。

### 4.2 PP-Structure 返回什么

PP-Structure 返回的是结构化结果，每个元素都有类型。

示例结构：

```python
[
    {
        "type": "title",
        "bbox": [100, 36, 500, 74],
        "res": [
            {
                "text": "印刷企业管理规范",
                "confidence": 0.9823,
                "text_region": [[102, 38], [498, 36], [498, 72], [102, 74]]
            }
        ]
    },
    {
        "type": "table",
        "bbox": [50, 220, 800, 520],
        "res": {
            "html": "<table><tr><td>材料</td><td>克重</td></tr></table>",
            "cells": [
                {"text": "材料", "confidence": 0.9512, "row": 0, "col": 0},
                {"text": "克重", "confidence": 0.9486, "row": 0, "col": 1}
            ]
        }
    }
]
```

常见字段：

| 字段 | 含义 |
| --- | --- |
| `type` | 元素类型，如 `title`、`text`、`table`、`figure` |
| `bbox` | 当前版面区域的边界框 |
| `res` | 当前区域的识别结果 |
| `text` | OCR 识别出的文字 |
| `confidence` | 文本行或表格单元格的识别置信度 |
| `html` | 表格转出的 HTML |
| `row` / `col` | 表格单元格所在行列 |

所以 PP-Structure 更适合复杂文档，因为它不只是识别文字，还保留了文档结构。

## 5. 质量把控指标

解析完成后，不是直接入库，而是先计算质量指标。

系统可以从 PP-Structure 返回结果中提取所有文本块和表格单元格的置信度，再结合文档结构计算综合质量分。

| 指标 | 权重 | 作用 |
| --- | --- | --- |
| 平均置信度 | 30% | 判断整体 OCR 识别质量 |
| 最低置信度 | 20% | 发现极低质量文本块 |
| 低置信度占比 | 15% | 统计低于阈值的文本比例 |
| 文本密度 | 15% | 判断是否存在空白页、漏识别页 |
| 结构完整性 | 10% | 判断是否识别出标题、正文、表格等结构 |
| 置信度稳定性 | 10% | 判断页面之间、文本块之间质量波动 |

其中低置信阈值可以先按 `0.7` 做演示口径，真实上线时应结合文档类型和人工抽检结果调整。

## 6. A/B/C/D 分级处理策略

综合评分后，可以把文档分为四级处理：

| 等级 | 分数范围 | 处理策略 |
| --- | --- | --- |
| A 级 | `>= 0.9` | 质量优秀，直接入库 |
| B 级 | `0.7 - 0.9` | 可以入库，但标记为待抽查 |
| C 级 | `0.5 - 0.7` | 不直接入库，触发降级处理或局部重解析 |
| D 级 | `< 0.5` | 拒绝入库，进入人工审核队列 |

这个分级的意义是：不要让低质量 OCR 结果污染知识库。因为一旦错误文本进入 Milvus，后面 RAG 检索和大模型生成都会受到影响。

## 7. 降级处理机制

C 级或部分页面低质量时，不一定要整篇文档重来，可以做更细的处理。

### 7.1 逐页定位问题

先按页统计质量分，找出低质量页。

例如第 3 页和第 8 页置信度低，就只对这两页重新处理，而不是整份文档重新 OCR。

可调整的处理方式包括：

- 提高图片 DPI 后重新 OCR。
- 对图片做去噪、增强、矫正。
- 对旋转或倾斜页面做方向校正。
- 对表格页单独启用表格结构识别。

### 7.2 引擎兜底

如果 PP-Structure 解析失败，可以降级到纯 PaddleOCR 或其他 OCR 引擎做文字兜底。

但兜底结果要标记解析方式，例如：

```text
parse_method = "paddleocr_fallback"
quality_level = "C"
need_review = true
```

这样后续排查召回错误时，可以知道这个 chunk 来自低质量兜底解析。

### 7.3 人工复核

所有引擎都无法稳定解析，或者涉及关键业务参数时，需要进入人工复核队列。

需要提供给人工复核人员的信息包括：

- 原始文件。
- 问题页码。
- 解析后的文本。
- 低置信文本块。
- 解析方式。
- 质量评分。
- 是否包含尺寸、材料、工艺参数、质量标准等高风险内容。

## 8. 入库元数据如何设计

为了保证可追溯，每个 chunk 入库时都要携带元数据。

| 字段 | 作用 |
| --- | --- |
| `doc_id` | 文档唯一 ID |
| `file_name` | 原始文件名 |
| `source` | 知识域，如印前规范、材料工艺、质量异常 |
| `page` | 页码 |
| `doc_type` | PDF、Word、PPT、图片、FAQ 表格等 |
| `parse_method` | `pdfplumber`、`pp_structure`、`paddleocr_fallback` 等 |
| `quality_score` | 综合质量分 |
| `quality_level` | A/B/C/D |
| `ocr_confidence` | 当前 chunk 或页面聚合后的 OCR 置信度 |
| `need_review` | 是否需要人工复核 |
| `version` | 文档版本 |
| `updated_at` | 文档更新时间 |

这里的 `ocr_confidence` 不是 OCR 天然返回的“整篇文档唯一分数”，而是系统根据文本块或单元格置信度聚合出来的质量字段。

## 9. 清洗、去重、脱敏和切块

质量评分通过后，还要继续做入库前治理。

### 9.1 清洗

清洗主要处理：

- 多余空格和异常换行。
- 页眉页脚。
- 重复目录。
- OCR 乱码。
- 标点缺失。
- 表格转 Markdown 后的格式整理。

### 9.2 去重

企业资料经常存在多个版本，例如“印前规范旧版”和“印前规范 2026 版”同时存在。

去重时要处理：

- 完全重复文件。
- 同一文档不同版本。
- 多份资料中重复出现的 SOP 片段。

版本冲突时，不应该简单都入库，而是保留最新版，旧版进入归档或标记不可检索。

### 9.3 脱敏

入库前需要脱敏：

- 客户名称。
- 订单号。
- 联系方式。
- 报价金额。
- 内部人员姓名。
- 其他不适合进入知识库的敏感信息。

### 9.4 父子块切分

切块不是固定长度一切到底，而是父子块结合：

- **父块**：保留章节、页面、表格或完整段落，用于给大模型提供完整上下文。
- **子块**：更短、更聚焦，用于提高向量检索命中率。

对表格、尺寸标准、材料规格、工艺参数等内容，尽量整体保留，避免把“条件、参数、结论”拆散。

## 10. 监控告警和持续优化

文档入库质量还需要长期监控。

可以监控这些指标：

| 指标 | 用途 |
| --- | --- |
| OCR 成功率 | 判断解析链路是否稳定 |
| 平均质量分 | 观察整体文档质量 |
| C/D 级文档比例 | 判断低质量资料是否增多 |
| 人工队列数量 | 判断复核是否堆积 |
| 低置信页面数量 | 定位扫描质量差的资料 |
| 入库失败数量 | 发现解析异常或格式异常 |

告警示例：

- 连续多个文档评分低于阈值，提示检查文档来源。
- 人工复核队列超过阈值，提示管理员处理。
- 某类文档 OCR 成功率突然下降，检查是否出现新的扫描格式。

## 11. 面试回答模板

### 11.1 3 分钟版

> 我们的文档入库不是简单把 PDF 切块后写入向量库，而是有一套离线质量把控流程。  
> 首先会根据文件类型选择解析方式：FAQ 表格走 `pandas` 或 `openpyxl`，普通文本 PDF 走 `pdfplumber`，扫描件和复杂图文资料走 PP-Structure。PP-Structure 的优势是既能做版面分析，也能返回每个文本块或表格单元格的置信度。
>
> 解析后不会直接入库，而是做质量评分。评分主要看 6 个维度：平均置信度、最低置信度、低置信度占比、文本密度、结构完整性和置信度稳定性。综合分数会分成 A/B/C/D 四级：A 级直接入库，B 级入库但标记抽查，C 级触发局部重解析或引擎兜底，D 级拒绝入库并进入人工审核。
>
> 降级处理时，我们不会整篇文档盲目重跑，而是先定位低质量页面，比如只对第 3 页、第 8 页提高 DPI、做图像增强后重新 OCR。如果还是不稳定，就降级到纯 PaddleOCR 兜底，并标记 `parse_method` 和 `need_review`。涉及尺寸、材料、工艺参数、质量标准的内容，会进入人工复核，避免错误知识污染 RAG 检索结果。
>
> 最后，入库 chunk 会保留 `doc_id`、页码、解析方式、质量分、OCR 置信度、版本号等元数据。这样线上如果出现错误回答，可以追溯到具体文档、具体页面和具体解析方式。

### 11.2 1 分钟版

> 文档入库我会先做格式识别，再解析、OCR、质量评分、清洗、脱敏、切块和向量化。  
> 扫描件和复杂图文资料用 PP-Structure，因为它不仅能识别文字，还能返回标题、正文、表格等结构，并且每个文本块有置信度。  
> 我们基于平均置信度、最低置信度、低置信度占比、文本密度、结构完整性和稳定性做综合评分，分成 A/B/C/D 四级。A 级直接入库，B 级抽查，C 级局部重解析或引擎兜底，D 级进入人工审核。  
> 核心原则是低质量 OCR 结果不能直接污染知识库，并且每个 chunk 都要能追溯来源、页码、解析方式和质量分。

## 12. 面试官可能追问

### 问：PP-Structure 和 PaddleOCR 的区别是什么？

> PaddleOCR 主要返回文字框、文字和置信度；PP-Structure 在 PaddleOCR 基础上增加了版面分析和表格识别。  
> 所以 PP-Structure 能告诉我们这个区域是标题、正文、表格还是图片，表格还可以返回 HTML 和单元格行列信息，更适合复杂文档入库。

### 问：OCR 会有一个总置信度吗？

> 通常不是天然只有一个总置信度。OCR 返回的是每个文本框或文本行的置信度。系统里的 `ocr_confidence` 一般是聚合值，比如当前页面或当前 chunk 的平均置信度、最低置信度，或者低置信文本占比。

### 问：如果 OCR 把关键参数识别错了怎么办？

> 涉及尺寸、材料、工艺参数、质量标准这类高风险内容，不会只依赖 OCR。低置信内容会进入人工复核；即使置信度较高，也会对关键规范文档做抽样检查。这样可以降低错误参数进入 RAG 知识库的风险。

### 问：为什么不直接用 MinerU？

> MinerU 适合复杂文档理解，但如果面试重点是入库质量把控，我会优先强调 PP-Structure，因为它能返回更直接可用的 OCR 置信度，方便做评分、分级和复核。实际工程里也可以混合使用：PP-Structure 做主解析和质量评分，疑难复杂文档再尝试其他解析工具兜底。

### 问：质量评分通过后就一定没问题吗？

> 不是。质量评分只能说明解析结果在 OCR 置信度和结构完整性上达标，不能保证业务内容一定正确。所以还要结合文档版本管理、关键资料人工确认、RAG 生成后的答案一致性校验一起控制风险。

## 13. 最终总结

文档入库质量把控可以概括成一句话：

> 入库前用 PP-Structure 拿到结构和置信度，基于多维指标做 A/B/C/D 分级；低质量内容不直接入库，而是局部重解析、引擎兜底或人工复核；入库后的每个 chunk 保留来源、页码、解析方式和质量分，确保错误可追溯、可下架、可重建。
