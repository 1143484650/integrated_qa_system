# 印刷企业内部知识库问答系统：精简包装大纲

## 1. 项目定位

本项目是面向印刷企业内部员工的知识库问答系统，主要服务客服、印前、售后、生产支持和知识库维护人员。

系统采用“标准问答优先 + 查询分类 + RAG 检索增强 + 大模型生成”的混合问答架构。高频稳定问题优先从 MySQL 标准问答库返回；未命中标准问答时，通过 BERT 分类判断是否属于印刷相关问题；专业问题进入 RAG 链路，从企业内部文档中检索依据，再由大模型生成回答。

系统定位是内部知识辅助工具，不直接参与报价、订单审核、责任判定、赔付结论、生产排程等业务决策。

## 2. 业务背景

印刷企业的知识通常分散在 FAQ、印前规范、材料工艺手册、售后 SOP、质量异常案例和历史问答中。员工遇到问题时，经常需要翻文档、问老员工或查历史记录，效率低且回答不统一。

本系统要解决的核心问题：

- 资料分散，检索成本高。
- 不同员工对同一问题的解释不一致。
- 新员工学习印前规范、材料工艺、质量异常处理成本高。
- 文档更新后，一线员工无法及时使用新知识。
- 复杂质量问题需要综合多个文档片段，普通关键词搜索效果有限。

## 3. 目标用户与典型价值

| 用户角色 | 主要需求 | 系统价值 |
| --- | --- | --- |
| 客服人员 | 快速解释印前、材料、工艺常见问题 | 提升答复一致性，减少反复咨询 |
| 印前人员 | 查询出血、CMYK、字体转曲、刀模线等规范 | 快速定位规范依据 |
| 售后人员 | 查询色差、覆膜起泡、胶装掉页等异常原因 | 获取排查方向和沟通参考 |
| 生产支持人员 | 查询材料、工艺和产品品类知识 | 辅助理解工艺注意事项 |
| 知识库维护人员 | 维护 FAQ、SOP、工艺手册和质量案例 | 保证知识更新进入检索链路 |

## 4. 系统边界

### 包含范围

- 内部知识问答。
- 高频 FAQ 标准回答。
- BERT 查询分类。
- RAG 文档检索增强。
- 知识域过滤和权限隔离。
- 多轮会话历史。
- WebSocket 流式输出。
- 文档解析、OCR、清洗、脱敏、切块和向量化入库。
- 印前规范、材料工艺、质量异常处理建议三个核心场景。

### 不包含范围

- 不自动报价。
- 不做订单预审。
- 不自动分单。
- 不控制生产设备。
- 不做生产排程。
- 不做质量责任判定。
- 不替代印前、工艺、质检、售后人员做最终结论。

## 5. 核心业务场景

### 5.1 印前文件规范查询

典型问题：

- 为什么印刷文件要留出血？
- RGB 文件直接印刷会有什么问题？
- 图片分辨率过低会影响什么？
- 字体为什么要转曲？
- 刀模线文件需要注意什么？

处理方式：

- 高频稳定问题进入 MySQL 标准问答库。
- BM25 高置信命中后直接返回标准答案。
- 未命中时进入 BERT 分类，判断是否需要进入 RAG。

### 5.2 材料与工艺知识查询

典型问题：

- 铜版纸和哑粉纸有什么区别？
- 白卡纸适合做哪些印刷品？
- 覆膜和上光有什么区别？
- 烫金适合哪些产品？

处理方式：

- 材料说明、工艺手册和产品知识文档进入 RAG 文档库。
- 使用 Milvus 混合检索召回相关片段。
- 大模型基于检索上下文组织回答。

### 5.3 质量异常处理建议查询

典型问题：

- 印刷色差常见原因有哪些？
- 覆膜起泡可能是什么原因？
- 胶装掉页一般怎么排查？
- 模切偏位可能和哪些环节有关？

处理方式：

- 系统只提供可能原因、排查方向和沟通参考。
- 涉及责任归属、赔付、客户争议时，提示人工确认。
- 回答中区分“可能原因”“排查方向”“需人工确认”。

## 6. 总体架构

系统分为三条主线：

### 6.1 离线知识入库流程

```text
数据源
  -> 文档解析 / OCR
  -> 清洗、去重、脱敏
  -> 父块结构切分
  -> 子块语义切分
  -> BGE-M3 向量化
  -> 写入 Milvus
```

主要数据源：

- FAQ Excel / CSV。
- 文本型 PDF。
- 复杂版式 PDF。
- 扫描件 PDF。
- Word、PPT、表格、SOP、工艺手册、质量案例。

### 6.2 在线问答流程

```text
用户提问
  -> FastAPI / WebSocket 接收请求
  -> 初始化 GraphState
  -> Redis 缓存查询
  -> MySQL + BM25 标准问答匹配
  -> BERT 查询分类
  -> 印刷关键词复核
  -> 检索策略选择
  -> Milvus dense + sparse 混合检索
  -> RRF 排名融合
  -> 父子块回溯
  -> CrossEncoder 精排
  -> 大模型生成草稿
  -> Faithfulness 轻量校验
  -> 风险控制
  -> 组装 citations
  -> WebSocket 返回最终答案
  -> MySQL 保存会话历史
```

### 6.3 辅助能力

- 检索调试接口：查看召回 chunk、dense/sparse/rerank 分数、权限过滤结果和最终 prompt 上下文。
- 离线评估：使用测试集和 RAGAS 指标评估检索与生成质量。
- 运行监控：记录 trace_id、慢请求、错误率、拒答率、超时率。

## 8. 知识库建设

### 8.1 入库流程

```text
上传文档
  -> 判断文档类型
  -> 选择解析方式
  -> 版面结构识别 / OCR
  -> 内容清洗
  -> 敏感信息脱敏
  -> 文档质量检测
  -> 父块切分
  -> 子块切分
  -> 向量化
  -> 写入 Milvus
```

### 8.2 文档解析策略

- 文本型 PDF：优先使用 pdfplumber。
- 复杂版式 PDF：使用 PP-Structure 解析标题、段落、表格和版面结构。
- 扫描件 PDF：使用 PP-Structure + PaddleOCR。
- FAQ 表格：使用 pandas / openpyxl 解析后写入 MySQL。

### 8.3 父子块切分

系统采用父子块结构：

- 子块：用于向量检索，粒度更细，便于精准召回。
- 父块：用于补全上下文，保证大模型生成时拿到完整语义。

典型流程：

```text
文档章节
  -> 父块：按标题、章节、页码、段落结构切分
  -> 子块：在父块内部做语义切分
  -> 子块入 Milvus
  -> 检索命中子块后回溯父块
```

这样可以兼顾召回精度和上下文完整性。

### 8.4 元数据设计

每个 chunk 需要保存：

- `chunk_id`
- `parent_id`
- `file_name`
- `source`
- `section`
- `page`
- `text`
- `created_at`
- `version`
- `ocr_confidence`

其中 `source` 用于知识域过滤，`parent_id` 用于父子块回溯，`version` 用于文档更新和缓存失效。

## 9. 查询路由与 RAG 设计

### 9.1 标准问答优先

高频稳定问题不走大模型，优先从标准问答库返回。

设计原因：

- 标准答案更稳定。
- 响应更快。
- 成本更低。
- 可以减少 RAG 幻觉风险。

只有 BM25 高置信命中时才返回标准答案，避免“相似但不准确”的误命中。

### 9.2 BERT 查询分类

标准问答未命中后，进入 BERT 分类：

- 印刷相关：进入 RAG。
- 明显无关：进入印刷关键词复核。
- 低置信：保守进入 RAG，避免误拦截专业问题。

关键词复核用于兜底，例如 BERT 判为无关，但问题中包含“出血、覆膜、CMYK、模切、色差”等领域词，则仍进入 RAG。

### 9.3 检索策略选择

根据问题类型选择不同检索策略：

- 直接检索：普通独立问题。
- HyDE：问题短、表达模糊时，先生成假设答案再检索。
- 回溯问题：多轮追问中，将“这个怎么处理”改写为完整问题。
- 子查询：复杂问题拆成多个子问题分别检索。

### 9.4 混合检索

RAG 检索采用 dense/sparse 混合召回：

```text
query
  -> BGE-M3 dense 向量
  -> BGE-M3 sparse 向量
  -> Milvus Hybrid Search
  -> dense top-K + sparse top-K
  -> RRF 融合
  -> 候选子块
  -> 回溯父块
  -> CrossEncoder 精排
  -> top-3 父块进入生成
```

设计原因：

- dense 检索擅长语义相似。
- sparse 检索擅长关键词、术语和工艺名匹配。
- RRF 避免两路分数尺度不一致。
- reranker 提升最终进入 prompt 的上下文质量。

### 9.5 重排序耗时优化

CrossEncoder 精排能提升准确率，但耗时和候选数量、文本长度强相关。

优化策略：

- 不是所有请求都走 rerank，FAQ 命中和无关问题直接提前结束。
- 只对 RRF 融合后的少量候选精排，例如 top-10 或 top-15。
- 不拿完整父块直接精排，优先用子块、标题或摘要打分，精排后再回溯父块。
- 限制 rerank 输入长度，例如 256 或 384 tokens。
- 对 rerank 设置独立超时，例如 800ms 到 1s。
- 超时后降级使用 RRF 排序结果。
- 高频 query 的 rerank 结果按 `query + final_sources + knowledge_version` 缓存。
- 服务启动时预加载 reranker，并做 warmup，避免首个请求冷启动。

面试表达：

> 重排序模型慢主要是因为 CrossEncoder 要对 query 和每个候选文档成对编码。我的处理不是去掉 rerank，而是在混合召回和 RRF 融合后，只保留少量候选进入精排；同时用短文本参与打分，精排后再回溯父块给大模型。工程上还会设置 rerank 超时和降级策略，超时就使用 RRF 结果，保证系统响应时间可控。

## 10. 答案生成与引用

### 10.1 生成策略

RAG 链路中，大模型只基于检索上下文回答，不允许脱离企业知识库自由发挥。

生成阶段输入：

- 用户当前问题。
- 最近会话历史。
- rerank 后的 top-3 父块。
- 回答约束和风险提示。

RAG 答案先在服务端生成草稿，不直接流式下发。只有通过一致性校验和风险控制后，才通过 WebSocket 返回最终答案。

### 10.2 引用来源

最终答案携带 citations，说明答案依据来自：

- 知识域。
- 文件名。
- 页码。
- 章节。
- chunk 片段。
- rerank 分数。

引用不是额外再检索一次，而是从本轮进入 prompt 的父块和子块中提取，避免引用和实际生成上下文不一致。

## 11. Faithfulness 一致性校验

RAG 生成后的答案需要检查是否被检索上下文支持。

在线链路采用轻量化校验，不额外调用 LLM Judge，主要使用：

- 关键词和实体覆盖。
- 新增数字、规格、参数检查。
- 答案关键句与 top-3 父块的 BGE-M3 相似度。
- 高风险问题的严格阈值。

处理规则：

- 有依据内容：保留。
- 无依据内容：删除。
- 缺少关键依据：提示“当前资料未明确说明，建议人工确认”。
- 责任判定、赔付、最终生产方案等高风险内容：转人工确认。

系统不判断现实世界的绝对真相，只判断“答案是否被当前检索上下文支持”。

## 12. 权限控制与知识域隔离

系统采用权限分离设计：

- 原印刷系统负责用户登录、组织架构、角色权限和 `allowed_sources` 计算。
- 知识库问答模块只解析 Token，并执行知识域过滤。

最终检索范围：

```text
final_sources = allowed_sources ∩ requested_sources
```

两次过滤：

1. Milvus 检索时使用 metadata filter：`source in final_sources`。
2. 检索结果返回后，再次校验每个 chunk 的 `source`。

这样可以防止 Milvus filter 失效或 rerank 引入越权内容。

## 13. 会话历史与多轮追问

系统通过 MySQL 保存会话历史，默认保留最近 5 轮。

使用方式：

- 生成阶段始终带入最近历史，帮助模型理解追问。
- 检索阶段默认优先使用当前问题，避免历史噪声干扰。
- 如果判断为追问，则通过回溯问题策略改写为完整独立问题后再检索。

示例：

```text
上一轮：覆膜起泡可能是什么原因？
当前轮：怎么排查？
改写后：覆膜起泡应该怎么排查？
```

## 14. 可靠性设计

### 14.1 知识来源可靠

- FAQ、规范、SOP、质量案例由业务人员维护。
- OCR 结果保留置信度，低置信内容进入复核。
- 客户名称、订单号、联系方式、报价金额等敏感信息入库前脱敏。
- 文档记录版本和更新时间。

### 14.2 查询路由可靠

- 标准问答只在高置信命中时返回。
- BERT 低置信结果保守进入 RAG。
- BERT 判无关后增加印刷关键词复核。
- 专业问题优先基于企业内部文档回答。

### 14.3 检索可靠

- 使用 dense/sparse 混合检索。
- 使用 RRF 融合两路结果。
- 使用父子块补全上下文。
- 使用 reranker 精排最终候选。
- 检索调试接口可观察 source 过滤、召回分数、rerank 分数和进入 prompt 的上下文。

### 14.4 生成可靠

- prompt 要求只基于上下文回答。
- 检索不到明确依据时拒答或提示人工确认。
- 生成后做 Faithfulness 校验。
- 高风险业务问题不做最终结论。

## 15. 异常处理与降级

| 组件 | 故障场景 | 降级策略 |
| --- | --- | --- |
| Redis | 连接失败 | 跳过缓存，直接查 MySQL 或 Milvus |
| MySQL | 标准问答查询失败 | 跳过标准问答，进入 RAG |
| BERT 分类器 | 模型不可用 | 默认进入 RAG |
| Milvus | 检索失败或超时 | 返回检索服务不可用提示 |
| Reranker | 超时或异常 | 使用 RRF 结果继续生成 |
| 大模型 | 超时或限流 | 重试 1 次，失败后返回生成失败 |

建议超时设置：

| 环节 | 超时时间 |
| --- | --- |
| Redis | 1 秒 |
| MySQL | 3 秒 |
| Milvus | 5 秒 |
| Reranker | 800ms 到 1 秒 |
| 大模型 | 30 秒 |
| 整体请求 | 60 秒 |

## 17. 面试讲述重点

### 17.1 项目一句话

这是一个面向印刷企业内部员工的知识库问答系统，通过 FAQ 标准问答、BERT 查询分类、Milvus 混合检索、CrossEncoder 精排和大模型生成，帮助员工快速查询印前规范、材料工艺和质量异常处理知识。

### 17.2 为什么不用纯大模型

纯大模型不能保证回答基于企业内部最新规范，容易出现幻觉，也无法做权限隔离和来源追溯。所以系统采用 RAG，让大模型基于企业知识库检索结果回答，并通过 citations 和 Faithfulness 校验保证可追溯、可控制。

### 17.3 为什么要标准问答优先

高频问题答案稳定，用标准问答更快、更准、成本更低。只有标准问答未命中时才进入 RAG，可以降低延迟和生成风险。

### 17.4 为什么要混合检索

印刷领域有大量专业术语，如 CMYK、出血、覆膜、模切、烫金。纯向量检索可能语义相似但漏掉精确术语，纯关键词检索又不理解表达变化。dense + sparse 混合检索可以同时兼顾语义召回和关键词匹配。

### 17.5 为什么要父子块

子块适合检索，粒度细，召回精准；父块适合生成，保留完整上下文。检索命中子块后回溯父块，可以避免只给模型零散片段导致回答不完整。

### 17.6 如何控制幻觉

主要通过四层控制：

- 检索阶段只从用户有权限的知识域召回。
- 生成 prompt 限制模型只能基于上下文回答。
- 生成后做 Faithfulness 校验，删除无依据内容。
- 高风险问题转人工确认，不输出责任判定或赔付结论。

### 17.7 如何处理重排序模型慢

不取消 rerank，而是控制它的输入规模和运行时间：

- FAQ 命中直接返回，不走 rerank。
- RRF 后只保留 top-10 或 top-15 候选。
- 用子块、标题或摘要参与 rerank，不直接精排长父块。
- 精排后再回溯父块给大模型。
- 设置 1 秒左右超时，超时降级到 RRF 结果。
- 高频问题缓存 rerank 结果。
- 服务启动时预加载模型并 warmup。

### 17.8 项目难点

- 多来源文档解析和 OCR 质量控制。
- 印刷专业术语下的召回准确率。
- 父子块切分粒度设计。
- reranker 精度和延迟平衡。
- 生成答案的依据一致性校验。
- 权限过滤和知识域隔离。
- 高风险业务问题的边界控制。
