# 印刷企业内部知识库问答系统大致流程

## 1. 项目定位

本项目可以包装为：**基于 RAG 的印刷企业内部知识库问答系统**。

它面向印刷企业内部员工使用，主要服务对象包括客服、印前人员、售后人员和生产支持人员。系统不参与报价、订单预审、自动分单或生产决策，而是把企业内部 FAQ、印前规范、材料工艺手册、质量异常处理文档等资料统一接入，形成一个可检索、可对话的内部知识库。

项目核心价值是解决内部知识分散、查找成本高、回答口径不一致的问题。过去员工需要从 PDF、Word、PPT、SOP 文档或历史问答中翻找资料，现在可以直接向系统提问。系统优先返回标准问答库中的稳定答案，如果标准问答无法覆盖，再从企业内部文档中检索相关内容，并结合大模型生成回答。

一句话概括：

```text
标准问答优先 + RAG 文档检索兜底 + 大模型组织回答 + 流式对话体验
```

## 2. 三类业务场景

为了避免项目包装过大，当前只突出三个内部知识查询场景。

### 2.1 印前文件规范查询

印前规范属于印刷企业内部高频知识。客服、设计对接人员和印前人员经常需要解释文件为什么要按规范提交。

典型问题包括：

- 为什么印刷文件要留出血？
- RGB 文件直接印刷会有什么问题？
- 图片分辨率太低会影响什么？
- 字体为什么要转曲？
- 刀模线文件需要注意哪些事项？

这个场景适合放入标准问答库，也适合接入印前检查规范、设计交付说明、文件制作要求等文档。高频问题可以通过 MySQL 标准问答快速返回，较复杂的问题则进入 RAG 检索印前规范文档。

### 2.2 材料与工艺知识查询

材料和工艺是印刷企业内部最常见的知识查询内容。不同纸张、不同印后工艺适合的产品和注意事项不同，员工在对外沟通前需要先理解基本区别。

典型问题包括：

- 铜版纸和哑粉纸有什么区别？
- 白卡纸适合做哪些印刷品？
- 覆膜和上光有什么区别？
- 烫金适合哪些产品？
- 包装盒为什么需要压痕和模切？

这个场景主要依赖材料说明、工艺手册、产品知识文档等资料。系统通过向量检索找到相关文档片段，再由大模型组织成更容易阅读的回答。

### 2.3 质量异常处理建议查询

质量异常处理适合包装成内部知识参考场景。系统只提供排查方向和常见原因，不做责任判定，也不替代质检或售后人员的最终判断。

典型问题包括：

- 印刷色差常见原因有哪些？
- 覆膜起泡可能是什么原因？
- 胶装掉页一般怎么排查？
- 模切偏位可能和哪些环节有关？
- 不干胶标签不粘可能是什么原因？

这个场景可以接入售后处理 SOP、质量异常案例、常见问题说明等文档。员工提问后，系统返回可能原因、排查方向和沟通参考，帮助内部人员快速定位问题范围。

## 3. 系统大致流程

系统在线问答链路可以分成两段：先查标准问答库，再走 RAG 知识库。

```text
用户提问
  -> 前端或 API 接收问题
  -> 判断是否为问候语等固定模板问题
  -> MySQL 标准问答库 + BM25 相似问题匹配
  -> 高置信命中：直接返回标准答案
  -> 未命中：进入 RAG 检索增强流程
  -> Milvus 检索内部知识文档
  -> BGE-M3 dense/sparse 混合检索
  -> BGE-Reranker 对候选文档精排
  -> 拼接上下文和历史会话
  -> 调用大模型生成回答
  -> WebSocket 流式返回给前端
  -> 将本轮问答写入 MySQL 会话历史
```

### 3.1 标准问答优先

系统会先使用 MySQL 中的标准问答库处理高频、确定性问题，例如“什么是出血”“为什么要字体转曲”“覆膜和上光有什么区别”等。

BM25 负责计算用户问题和标准问题之间的相似度。如果分数超过阈值，说明用户问题和已有标准问答足够接近，系统直接返回标准答案。这样可以保证高频问题回答稳定、速度快，也减少不必要的大模型调用。

Redis 用于缓存问题列表、分词结果和高频答案，降低重复读取 MySQL 和重复计算的成本。

### 3.2 RAG 文档检索兜底

如果标准问答库没有可靠命中，系统进入 RAG 流程。RAG 的目标不是让大模型凭空回答，而是先从内部知识库中检索相关资料，再让大模型基于上下文组织答案。

RAG 流程包括：

1. 使用 BERT 查询分类器判断问题是否需要进入专业知识检索。
2. 使用策略选择器选择直接检索、子查询检索、HyDE 或回溯问题检索。
3. 使用 BGE-M3 生成 dense 向量和 sparse 向量。
4. 在 Milvus 中执行混合检索。
5. 使用 BGE-Reranker 对候选父文档进行精排。
6. 取最相关的文档片段拼接成上下文。
7. 将上下文、用户问题和最近会话历史交给大模型生成回答。

这种设计适合内部知识库场景，因为企业文档内容多、格式杂、查询问题也不固定。标准问答解决高频问题，RAG 负责覆盖更开放、更复杂的内部知识查询。

### 3.3 知识域过滤

现有系统中的 `source_filter` 可以包装为知识域过滤。员工可以选择某一类知识范围，减少无关文档召回。

示例知识域：

| 知识域 | 含义 | 对应场景 |
| --- | --- | --- |
| `prepress` | 印前文件规范 | 出血、CMYK、分辨率、字体转曲、刀模线 |
| `material_process` | 材料与工艺知识 | 纸张、覆膜、烫金、UV、模切、胶装 |
| `quality_issue` | 质量异常处理 | 色差、覆膜起泡、掉页、偏位、不粘 |

答辩时可以说明：演示阶段主要覆盖这三个知识域，后续如果企业有更多资料，也可以继续扩展其他内部知识来源。

## 4. 知识入库流程

在线问答依赖知识库中的文档向量，因此需要先把内部资料处理成可检索数据。

可接入的内部资料包括：

- 高频 FAQ 和标准问答。
- 印前文件规范。
- 材料与工艺说明文档。
- 质量异常处理说明。
- 售后处理 SOP 或内部沟通参考。

文档入库流程如下：

```text
内部资料
  -> 文档加载器读取 PDF / Word / PPT / 图片 / TXT
  -> OCR 提取扫描件或图片中的文字
  -> 添加 source、file_path、timestamp 等元数据
  -> 父块切分，保留较完整上下文
  -> 子块切分，提高检索命中精度
  -> BGE-M3 生成 dense/sparse 向量
  -> 写入 Milvus 向量库
```

项目采用父子块切分思路：子块适合检索，粒度更细，更容易命中用户问题；父块适合生成，内容更完整，可以给大模型提供足够上下文。检索时先命中子块，再回到父块参与回答生成。

## 5. 边界说明

这个项目只做内部知识查询，不扩大成业务决策系统。

明确不包含：

- 不自动报价。
- 不做订单预审。
- 不做自动分单。
- 不控制印刷设备。
- 不做生产排程。
- 不做质量责任判定。

对于质量异常类问题，系统只返回常见原因、排查方向和内部处理建议，最终判断仍由质检、售后或相关负责人完成。

这样包装的好处是边界清晰：另一个项目可以负责订单预审或报价前处理，这个项目只负责内部知识服务，两者不会混在一起。

## 6. 答辩讲解版本

### 6.1 一分钟项目介绍

> 我这个项目可以理解为一个面向印刷企业内部员工的知识库问答系统。印刷企业内部有大量资料分散在 FAQ、印前规范、材料工艺手册和质量异常处理文档里，员工遇到问题时需要反复翻文档，查询成本高，回答口径也不容易统一。
>
> 所以我做的是一个“标准问答优先 + RAG 文档检索兜底”的系统。对于高频确定性问题，系统先用 MySQL 标准问答库和 BM25 相似匹配快速返回稳定答案；如果标准问答没有命中，就进入 RAG 流程，从 Milvus 向量库中检索内部文档，经过 BGE-M3 混合检索和 Reranker 精排后，再调用大模型生成回答。系统支持知识域过滤、会话历史和 WebSocket 流式输出，主要覆盖印前文件规范、材料与工艺知识、质量异常处理建议三个内部场景。

### 6.2 流程讲解话术

> 用户提问后，系统不会一上来就直接调用大模型，而是先查标准问答库。比如“为什么要字体转曲”这类高频问题，如果 BM25 能匹配到已有标准答案，就直接返回，这样稳定且速度快。
>
> 如果标准问答没有命中，系统再进入 RAG。RAG 会根据问题从内部知识文档中检索相关内容，例如印前规范、材料工艺手册或者质量异常处理文档。检索时使用 BGE-M3 同时生成稠密向量和稀疏向量，在 Milvus 中做混合检索，再用 Reranker 对候选文档排序，最后把最相关的上下文交给大模型生成答案。
>
> 最终答案通过 WebSocket 流式返回，用户可以像聊天一样逐步看到回答。系统还会保存最近几轮会话历史，支持连续追问。
