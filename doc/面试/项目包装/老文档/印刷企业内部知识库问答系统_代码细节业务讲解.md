# 印刷企业内部知识库问答系统：代码细节业务讲解

## 1. 讲解总原则

讲代码细节时，不需要每一句都强行绑定印刷业务。更自然的讲法是：

```text
业务背景说明为什么做
系统链路说明整体怎么流转
代码模块说明具体怎么实现
业务细节点用于解释设计价值
边界说明避免项目被问成过大系统
```

这个项目的核心定位是：**基于 RAG 的印刷企业内部知识库问答系统**。

它不是自动报价系统，不做订单预审，不做自动分单，也不参与生产排程或质量责任判定。它的作用是把印刷企业内部散落在 FAQ、印前规范、材料工艺手册、质量异常处理文档里的知识统一接入，让内部员工可以直接提问查询。

答辩时可以采用一句话口径：

> 业务上，我把它包装成印刷企业内部知识查询系统；技术上，它是一个标准问答优先、RAG 文档检索兜底，并支持流式输出和多轮上下文的混合问答系统。

## 2. 推荐讲解顺序

### 2.1 先讲业务背景

印刷企业内部有很多知识并不适合直接写死在系统里，例如：

- 印前文件规范：出血、CMYK、图片分辨率、字体转曲、刀模线。
- 材料与工艺知识：铜版纸、哑粉纸、覆膜、上光、烫金、UV、模切、压痕、胶装。
- 质量异常处理建议：色差、覆膜起泡、胶装掉页、模切偏位、不干胶不粘。

这些知识有两个特点：

- 高频固定问题适合做标准问答。
- 复杂问题往往分散在文档里，更适合通过 RAG 检索后生成回答。

所以系统采用“标准问答优先 + RAG 文档检索兜底”的设计。

### 2.2 再讲系统链路

可以按下面这条主线讲：

```text
用户提问
  -> FastAPI 接收请求
  -> 判断是否为问候语或固定模板问题
  -> MySQL 标准问答库 + BM25 相似问题匹配
  -> 高置信命中：直接返回标准答案
  -> 未命中：进入 RAG 流程
  -> Milvus 检索内部文档
  -> BGE-M3 混合检索召回候选片段
  -> BGE-Reranker 精排候选内容
  -> 结合历史会话和检索上下文调用大模型
  -> WebSocket 流式返回答案
  -> MySQL 保存最近会话历史
```

讲到这里时，业务只要点到为止：高频规范问题走标准问答，复杂工艺和异常问题走 RAG。

### 2.3 最后讲关键代码模块

代码细节讲解可以按层拆开，而不是按文件从头读到尾：

- 接口层：`app.py`
- 编排层：`new_main.py`
- 标准问答层：`mysql_qa/retrieval/bm25_search.py`
- RAG 检索层：`rag_qa`
- 会话历史与流式输出：MySQL `conversations` 表和 WebSocket

## 3. 接口层：`app.py`

### 3.1 `/api/query`：标准问答优先

`/api/query` 是非流式接口。它适合讲“先查标准问答库”的思路。

业务上可以举这些问题：

- 为什么印刷文件要留出血？
- RGB 文件直接印刷会有什么问题？
- 字体为什么要转曲？

这些问题答案相对固定，适合放在 MySQL 标准问答库里。系统先通过 BM25 匹配用户问题和标准问题，如果置信度足够高，就直接返回标准答案。

讲解重点：

- 不是所有问题都直接调用大模型。
- 高频、确定性强的问题优先走标准问答，速度更快，答案更稳定。
- 这类问题更适合统一企业内部口径。

### 3.2 `/api/stream`：复杂问题流式回答

`/api/stream` 是 WebSocket 接口，适合讲复杂问题的流式生成。

业务上可以举这些问题：

- 覆膜起泡可能是什么原因？
- 模切偏位一般和哪些环节有关？
- 不干胶标签不粘可能怎么排查？

这些问题通常不是一句标准答案能覆盖的，需要检索多个文档片段，再由大模型整理成结构化回答。因此系统通过 WebSocket 逐 token 返回，用户不用等完整答案生成完才看到结果。

讲解重点：

- WebSocket 主要改善长答案等待体验。
- RAG 回答可能较长，流式输出更像真实对话。
- 面试时不需要展开 WebSocket 协议细节，重点讲为什么这里适合流式。

### 3.3 `source_filter`：知识域过滤

代码里的 `source_filter` 可以包装成知识域过滤。

可以这样映射：

| 知识域 | 业务含义 | 适合问题 |
| --- | --- | --- |
| `prepress` | 印前文件规范 | 出血、CMYK、分辨率、字体转曲、刀模线 |
| `material_process` | 材料与工艺知识 | 纸张、覆膜、烫金、UV、模切、胶装 |
| `quality_issue` | 质量异常处理 | 色差、起泡、掉页、偏位、不粘 |

讲解重点：

- 用户选择知识域后，系统可以减少无关文档召回。
- 对内部知识库来说，不同部门关注的知识范围不同，过滤可以提升检索准确性。
- 这只是知识范围过滤，不是业务决策流程。

### 3.4 需要避开的细节

当前代码里的问候语回复仍然是“黑马程序员 / IT 教育”口径。面试讲印刷企业包装时，不建议主动展示这部分。

如果被问到，可以这样回答：

> 当前项目原始 Demo 里保留了一些通用问候模板，真正包装到印刷企业场景时，这部分会替换成企业内部助手的欢迎语。核心问答链路不依赖这部分。

## 4. 编排层：`new_main.py`

### 4.1 `IntegratedQASystem`：统一调度层

`IntegratedQASystem` 可以讲成系统的统一编排层。它负责初始化 MySQL、Redis、BM25、Milvus、RAGSystem 和大模型客户端，并控制一次提问的完整流转。

业务讲法：

> 用户提问后，系统不会直接让大模型自由发挥，而是先查企业内部标准问答库。如果能命中稳定答案，就直接返回；如果不能命中，再进入企业文档检索流程。

技术讲法：

- `bm25_search.search()` 负责标准问答命中。
- `rag_system.generate_answer()` 负责 RAG 检索增强回答。
- `call_dashscope()` 负责调用大模型并流式返回内容。
- `update_session_history()` 负责保存问答历史。

### 4.2 `threshold=0.85`：高置信命中才返回

代码里 BM25 匹配使用了 `threshold=0.85`。这个点很适合被业务化解释。

可以这样讲：

> 标准问答库适合回答确定性问题，但如果相似度不够高，强行返回标准答案反而会误导用户。所以这里设置了较高阈值，只有问题和标准问法足够接近时才直接返回，否则交给 RAG 去查更完整的内部文档。

印刷业务例子：

- “什么是出血”可以直接命中标准答案。
- “包装盒刀模线和出血怎么配合”更复杂，不应该只返回“什么是出血”的固定答案。

### 4.3 会话历史：支持连续追问

`conversations` 表保存 `session_id`、`question`、`answer` 和 `timestamp`，并保留最近 5 轮历史。

业务上适合这样讲：

> 内部员工查询时经常会连续追问。比如先问“覆膜起泡有哪些原因”，再问“那哑膜是不是更容易出现”，系统需要知道上一轮讨论的是覆膜异常，而不是把第二个问题当成孤立问题。

讲解重点：

- 最近 5 轮历史足够覆盖普通追问。
- 控制历史长度可以避免 prompt 过长。
- 历史只用于辅助上下文，不代表系统做生产判断。

## 5. 标准问答层：`mysql_qa/retrieval/bm25_search.py`

### 5.1 MySQL：保存高频标准问题

MySQL 适合保存高频、稳定、需要统一口径的 FAQ。

印刷场景里可以包装为：

- 出血是什么，通常为什么要留出血。
- RGB 和 CMYK 有什么区别。
- 图片分辨率过低会有什么影响。
- 字体转曲的目的是什么。
- 覆膜和上光的基本区别是什么。

讲解重点：

- 标准问答不是为了覆盖所有问题。
- 它主要解决高频问题的速度和稳定性。
- 这些答案可以由企业内部人员维护，保证口径统一。

### 5.2 BM25：适合行业术语匹配

BM25 是关键词相关性检索。它对印刷行业术语比较友好，因为很多问题会包含明确关键词。

例如：

- “烫金”
- “UV”
- “模切”
- “压痕”
- “覆膜”
- “不干胶”

讲解重点：

- BM25 不依赖大模型生成，速度快。
- 对专有名词、工艺术语、固定问法有优势。
- 它适合做标准问答库的第一层召回。

### 5.3 Redis：缓存高频答案

Redis 在这里主要用来缓存问题列表、分词结果和已命中的答案。

业务化讲法：

> 企业内部有些问题会被不同岗位反复问，比如“为什么要字体转曲”或者“覆膜和上光有什么区别”。这些问题命中后可以缓存起来，减少重复查库和重复计算。

讲解重点：

- Redis 是性能优化，不改变业务逻辑。
- 缓存适合高频查询。
- 数据最终来源仍然是 MySQL 标准问答库。

## 6. RAG 层：`rag_qa`

### 6.1 Milvus + BGE-M3：检索内部文档

RAG 层适合讲“内部文档知识检索”。

可以接入的印刷企业文档包括：

- 印前文件提交规范。
- 材料与工艺说明手册。
- 印后工艺注意事项。
- 售后处理 SOP。
- 质量异常案例和排查说明。

讲解重点：

- Milvus 负责存储和检索向量。
- BGE-M3 可以做 dense/sparse 混合检索。
- 混合检索比单纯关键词或单纯向量更适合复杂内部文档。

业务例子：

> 用户问“覆膜起泡怎么排查”时，问题可能同时涉及纸张含水率、覆膜温度、胶水、压力、存放环境等多个因素。标准 FAQ 很难完整覆盖，这时就需要从质量异常文档和工艺手册中检索相关片段。

### 6.2 Reranker：候选内容精排

Reranker 可以讲成第二道筛选。

第一阶段检索会召回一批可能相关的文档片段，但这些片段不一定都最适合回答当前问题。Reranker 会重新判断“问题和片段”的相关性，把更有价值的内容排到前面。

业务例子：

> 用户问“模切偏位和刀模线有关吗”，向量检索可能召回很多包含“模切”的内容，但真正有用的是讲刀模线、套准、纸张变形、设备定位的片段。Reranker 的作用就是把这些更贴近问题的内容排上来。

### 6.3 父子块切分：命中细，回答全

父子块切分可以这样讲：

- 子块：粒度小，适合检索命中。
- 父块：上下文更完整，适合交给大模型生成答案。

印刷文档里，经常一个规范点依赖上下文。例如讲“出血”时，可能同时涉及成品尺寸、裁切误差、安全边距和刀模线。如果只把一个短句交给大模型，回答可能不完整；如果直接用大块检索，又可能不容易精准命中。所以采用父子块思路比较合适。

## 7. 三类业务细节点如何穿插

### 7.1 印前文件规范

适合对应的技术点：

- MySQL 标准问答。
- BM25 相似问题匹配。
- 高频问题缓存。

可讲问题：

- 为什么要留出血？
- RGB 文件为什么不建议直接印刷？
- 字体为什么要转曲？
- 刀模线文件要注意什么？

边界口径：

> 系统只解释印前规范和注意事项，不自动检查用户上传文件，也不替代印前人员做最终文件审核。

### 7.2 材料与工艺知识

适合对应的技术点：

- RAG 文档检索。
- 知识域过滤。
- BGE-M3 混合检索。
- Reranker 精排。

可讲问题：

- 铜版纸和哑粉纸有什么区别？
- 覆膜和上光有什么区别？
- 烫金适合哪些产品？
- 包装盒为什么需要模切和压痕？

边界口径：

> 系统提供材料和工艺知识查询，不直接推荐最终生产方案，也不替代工艺工程师做确认。

### 7.3 质量异常处理建议

适合对应的技术点：

- 多轮会话历史。
- RAG 综合回答。
- WebSocket 流式输出。

可讲问题：

- 色差可能有哪些原因？
- 覆膜起泡怎么排查？
- 胶装掉页一般和哪些因素有关？
- 不干胶标签不粘可能是什么原因？

边界口径：

> 系统只提供常见原因和排查方向，不做质量责任判定，也不替代质检、售后或生产负责人给出最终结论。

## 8. 容易被追问的问题与回答口径

### 8.1 为什么不直接全部走大模型？

可以回答：

> 因为内部知识问答里有很多高频、稳定、需要统一口径的问题。全部走大模型会增加成本和延迟，也可能出现表达不稳定的问题。所以我先用 MySQL + BM25 处理标准问答，只有标准库覆盖不了的问题才进入 RAG。

### 8.2 为什么需要 RAG？

可以回答：

> 印刷企业内部资料很多，像材料工艺、质量异常、售后 SOP 这类内容会分散在不同文档里。RAG 的作用是先从企业内部文档中检索相关内容，再让大模型基于检索上下文组织回答，减少模型凭空回答的问题。

### 8.3 为什么需要 Reranker？

可以回答：

> 第一阶段检索主要负责召回，宁愿多召回一些相关候选；Reranker 负责把和当前问题最相关的片段排到前面。比如同样包含“模切”的文档很多，但用户问的是“模切偏位”，就应该优先保留和套准、刀模线、设备定位相关的片段。

### 8.4 为什么保留最近 5 轮历史？

可以回答：

> 多轮历史是为了支持连续追问，但历史太长会增加 prompt 长度，也可能引入无关上下文。最近 5 轮是一个折中，能覆盖大多数连续问答场景。

### 8.5 这个项目能不能做报价或订单预审？

可以回答：

> 这个项目我没有把它包装成报价或订单预审系统。它只负责内部知识查询，帮助员工快速查规范、工艺和异常处理知识。报价、订单预审、自动分单属于另一个业务系统的边界。

## 9. 讲解禁区

讲这个项目时，建议避免以下表达：

- 系统可以自动报价。
- 系统可以判断订单能不能生产。
- 系统可以自动分配订单到产线。
- 系统可以控制生产设备。
- 系统可以判定质量责任。
- 系统可以替代印前、工艺、质检或售后人员。

更稳妥的表达是：

- 系统提供内部知识查询。
- 系统统一高频问题回答口径。
- 系统辅助员工快速定位文档知识。
- 系统提供质量异常排查方向。
- 系统支持连续追问和流式回答。

## 10. 一段完整讲解示例

> 我这个项目业务上包装成印刷企业内部知识库问答系统，主要面向客服、印前、售后和生产支持人员。印刷企业内部有很多知识分散在 FAQ、印前规范、材料工艺手册和质量异常处理文档里，员工查询成本比较高，而且回答口径不一定统一。
>
> 所以系统采用了“标准问答优先 + RAG 文档检索兜底”的架构。用户提问后，接口层先接收请求，系统会优先使用 MySQL 中的标准问答库，通过 BM25 做相似问题匹配。像“为什么要留出血”“字体为什么要转曲”这种高频固定问题，如果匹配置信度超过阈值，就直接返回标准答案，并通过 Redis 缓存高频结果。
>
> 如果标准问答没有命中，就进入 RAG 流程。系统会从 Milvus 向量库里检索内部文档，使用 BGE-M3 做 dense 和 sparse 混合检索，再通过 Reranker 对候选片段精排，最后把相关上下文、用户问题和最近会话历史交给大模型生成回答。对于较长的回答，系统通过 WebSocket 流式返回，用户可以逐步看到结果。
>
> 在业务场景上，我主要覆盖三类内部知识：印前文件规范、材料与工艺知识、质量异常处理建议。项目边界上，它不做自动报价、订单预审和生产决策，只作为内部知识查询和排查辅助工具。
