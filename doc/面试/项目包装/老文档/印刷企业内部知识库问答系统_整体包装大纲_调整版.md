# 印刷企业内部知识库问答系统：项目设计与包装文档（调整版）

> 本文档基于实际代码实现进行大幅调整，核心目标是让包装叙事与可演示、可深挖的代码能力完全对齐，避免面试中“设计很先进、代码跟不上”的风险。

---

## 1. 项目概述

印刷企业内部知识库问答系统面向客服、印前、售后、生产支持和知识库维护人员，提供企业内部知识检索、标准问答、多轮追问和流式回答能力。

**系统定位**：内部知识辅助工具，采用“**标准问答优先 + BERT 分类路由 + 多策略 RAG 检索增强**”的混合架构。

- 高频、答案稳定的问题由 MySQL 标准问答库 + BM25 直接返回（高置信阈值控制）。
- 未命中或专业性问题，先通过 BERT 查询分类器判断是否需要查询企业知识库。
- 专业咨询类问题进入 RAG 链路：StrategySelector 选择检索策略 → BGE-M3 混合检索 → Reranker 精排 → 结合会话历史生成回答。
- 支持知识域过滤（source_filter）、最近 5 轮会话历史、WebSocket 流式输出。

系统严格限定为**知识查询辅助**，不执行报价、订单审核、生产排程、质量责任判定等业务决策。

---

## 2. 业务背景

印刷企业内部知识通常分散在 FAQ、印前规范、材料工艺手册、售后 SOP、质量异常案例和历史问答记录中。客服、印前、售后和生产支持人员遇到问题时，经常需要翻阅多个文档、询问经验员工或查询历史记录。

主要痛点：
- 资料分散（PDF、Word、PPT、扫描件、表格混杂）
- 标准不统一（不同员工解释差异大）
- 新员工学习成本高（印前规范、工艺、异常处理依赖经验）
- 复杂问题检索困难（质量异常往往需要综合多个文档片段）

国家新闻出版署《印刷业数字化三年行动计划（2025-2027年）》提出要完善印前-印刷-印后数据交换和标准体系、建设质检案例数据库。本系统聚焦企业内部知识服务，是印刷企业知识数字化的落地实践。

## 4. 系统范围

### 4.1 包含范围
- 内部知识问答
- 高频 FAQ 标准回答（MySQL + BM25 高置信命中）
- BERT 查询分类（区分通用知识 vs 专业咨询）
- 多策略 RAG 文档检索增强（直接检索 / 子查询 / 回溯问题 / HyDE）
- BGE-M3 混合检索 + BGE-Reranker 精排
- 知识域过滤（source_filter）
- 多轮会话历史（最近 5 轮）
- WebSocket 流式输出
- 文档解析、OCR、清洗、脱敏、父子块切分、向量化入库
- 印前规范、材料工艺、质量异常处理建议三个主要场景

### 4.2 不包含范围（必须在面试中主动说清边界）
- 不自动报价
- 不做订单预审
- 不做自动分单
- 不控制生产设备
- 不做生产排程
- 不做质量责任判定
- 不替代印前、工艺、质检、售后人员做最终结论
- 不直接处理客户订单交易数据

## 5. 核心业务场景与技术映射

### 5.1 印前文件规范查询
**典型问题**：
- 为什么印刷文件要留出血？
- RGB 文件直接印刷会有什么问题？
- 字体为什么要转曲？
- 图片分辨率过低会影响什么？

**处理方式**：
- 高频标准问题沉淀到 MySQL 标准问答库。
- BM25 相似问题匹配，高置信（threshold=0.85）直接返回标准答案。
- 未高置信命中 → BERT 分类 → 专业咨询则进入 RAG 检索印前规范文档。

**可靠性控制**：
- 标准问答由业务人员维护，BM25 设置高置信阈值，避免“看起来像但不准”的问题被直接返回。

### 5.2 材料与工艺知识查询
**典型问题**：
- 铜版纸和哑粉纸有什么区别？
- 白卡纸适合做哪些印刷品？
- 覆膜和上光有什么区别？
- 烫金适合哪些产品？

**处理方式**：
- 材料工艺手册、产品说明文档进入 Milvus 向量库。
- 专业问题通过 StrategySelector 选择检索策略（默认直接检索，复杂问题可走子查询/回溯/HyDE）。
- BGE-M3 生成 dense + sparse 向量 → Milvus 混合检索 → Reranker 精排 → 大模型基于上下文组织回答。

**可靠性控制**：
- 使用 `source_filter` 做知识域过滤（material_process）。
- 混合检索兼顾语义和关键词（行业术语如“覆膜”“烫金”）。
- Reranker 对候选内容做 query-doc 相关性重排。

### 5.3 质量异常处理建议查询
**典型问题**：
- 印刷色差常见原因有哪些？
- 覆膜起泡可能是什么原因？
- 胶装掉页一般怎么排查？
- 模切偏位可能和哪些环节有关？

**处理方式**：
- 接入售后 SOP、质量异常案例文档。
- 系统返回“可能原因 + 排查方向 + 建议人工确认”，不支持直接给出责任结论。
- 支持多轮追问（最近 5 轮历史拼接进 prompt）。

**可靠性控制**：
- 回答中明确区分“可能原因”“排查方向”“建议确认”。
- 检索不到明确依据时返回提示。
- 高风险业务问题倾向于保守回答或提示转人工。

## 6. 总体架构（基于实际实现）

### 6.1 实际运行架构（已落地）

```
用户提问
  ↓
FastAPI / WebSocket 接口层（app.py / api.py）
  ↓
IntegratedQASystem 统一调度（new_main.py）
  ├── MySQL + BM25 + Redis 标准问答（高置信 threshold=0.85 直接返回）
  │     └── 命中 → 直接返回 + 写入会话历史
  │
  └── 未命中 → RAGSystem（rag_qa/core/new_rag_system.py）
        ├── QueryClassifier（BERT 分类：通用知识 / 专业咨询）
        │     └── 通用知识 → 直接调用 LLM（不查企业知识库）
        │
        └── 专业咨询 → StrategySelector 选择策略
              ├── 直接检索 / 子查询检索 / 回溯问题检索 / HyDE
              ↓
              BGE-M3 dense + sparse 混合检索（Milvus）
              ↓
              hybrid_search_with_rerank（BGE-Reranker 精排父文档）
              ↓
              拼接检索上下文 + 最近 5 轮历史
              ↓
              大模型生成（DashScope / Qwen-Plus 兼容接口）
              ↓
              WebSocket 流式返回 + 写入 MySQL 会话历史
```

### 6.2 核心设计亮点（面试重点讲这几点）

1. **标准问答优先 + 高置信阈值**
   - 解决高频问题速度和一致性问题。
   - BM25 只做“是否足够像标准问题”的判断，不负责生成答案。

2. **BERT 查询分类作为智能闸门**
   - 在进入昂贵的 RAG 检索前，先判断“这是不是需要查企业内部知识的专业问题”。
   - 通用知识直接走大模型，减少无效检索和无关上下文干扰。
   - 低置信/边界问题保守走 RAG（避免专业问题被误判为通用）。

3. **StrategySelector 多策略检索**
   - 不是所有问题都用同一个检索方式。
   - 简单问题直接检索；复杂、模糊问题自动走子查询、回溯问题或 HyDE 生成假设答案再检索。
   - 提升复杂工艺和质量异常问题的召回质量。

4. **两阶段召回（混合检索 + Reranker）**
   - BGE-M3 同时支持语义召回和关键词召回（对印刷术语友好）。
   - Reranker 作为第二道筛选，把真正与当前问题相关的片段排到前面。

5. **父子块 + 知识域过滤 + 历史控制**
   - 子块用于精确命中，父块用于给大模型完整上下文。
   - `source_filter` 实现知识域隔离（prepress / material_process / quality_issue）。
   - 最近 5 轮历史控制 prompt 长度，同时支持连续追问。

### 6.3 与“理想架构”的差异说明（面试中主动说清）

设计阶段曾规划使用 LangGraph 进行更显式的状态机编排（包含生成后 Faithfulness 校验节点、服务端缓冲后再流式等），以进一步提升可靠性。

**当前实现采用更务实的过程式编排**（BERT 分类 + StrategySelector + 两阶段检索），已在核心链路上实现了分类路由、多策略检索、精排和历史感知，**可靠性通过阈值控制、保守路由、知识域过滤、Reranker 和 prompt 约束来保障**。

后续演进方向包括：引入 LangGraph 进行流程可视化编排、增加生成后一致性校验节点、构建文档入库任务队列等。

**这样讲的好处**：既展示了架构思考深度，又不会因为“设计很牛但代码里没有”被追问翻车。

| 分层       | 技术栈                                      | 作用 |
|------------|---------------------------------------------|------|
| 接口服务   | Python、FastAPI、WebSocket                  | 提供普通问答与流式问答 |

| 切块策略   | 父子块切分 + source/file_name/page/section 元数据 | 检索精度与生成上下文兼顾 |
| 向量化     | BGE-M3（dense + sparse）                    | 同时支持语义和关键词召回 |
| 向量库     | Milvus Hybrid Search + WeightedRanker       | 存储与混合检索 |
| 精排       | BGE-Reranker-large（CrossEncoder）          | query-doc 相关性重排 |
| 生成模型   | DashScope / Qwen-Plus（OpenAI-compatible）  | 基于上下文组织回答 |
| 会话存储   | MySQL（conversations 表，保留最近 5 轮）    | 支持多轮追问 |
| 评估       | RAGAS（Faithfulness、Answer Relevance 等）  | 离线评估检索与生成质量 |

## 8. 知识库建设

### 8.1 知识来源（按知识域拆分）

| 知识域            | 主要内容                     | 进入方式          | 典型问题场景         |
|-------------------|------------------------------|-------------------|----------------------|
| prepress          | 印前规范（出血、CMYK、转曲、刀模线等） | RAG 文档库        | 印前文件规范查询     |
| material_process  | 材料工艺手册（纸张、覆膜、烫金、模切等） | RAG 文档库        | 材料与工艺知识查询   |
| quality_issue     | 质量异常案例、售后 SOP       | RAG 文档库        | 质量异常处理建议查询 |
| faq（部分）       | 高频标准问题                 | MySQL 标准问答库  | 所有高频确定性问题   |

### 8.2 入库流程

```
内部资料收集
  → 按知识域分类（prepress / material_process / quality_issue）
  → 文档类型判断（PDF/Word/PPT/扫描件/Excel）
  → 解析（pdfplumber + 自定义 Edu Loader + OCR）
  → 文本清洗、去重、敏感信息脱敏
  → 补充元数据（source、file_name、page、department、version 等）
  → 父子块切分
  → BGE-M3 向量化
  → 写入 Milvus
```

高频、答案稳定的 FAQ 优先写入 MySQL 标准问答库，其余内容进入向量库。

### 8.3 元数据设计（实际使用）

- `source`：知识域过滤核心字段（prepress / material_process / quality_issue）
- `file_name`、`page`、`section`：可追溯性
- `doc_type`、`parse_method`、`ocr_confidence`：解析质量跟踪
- `department`、`version`、`update_time`：业务归属与版本管理

检索时通过 `source in final_sources` 做 metadata filter，实现知识域隔离。

## 9. 查询路由与 RAG 核心设计

### 9.1 标准问答优先（第一道防线）

- BM25 只负责相似问题匹配，不生成答案。
- 只有分数 ≥ 0.85 才返回标准答案，否则进入后续流程。
- 作用：保证高频问题速度快、口径统一、成本低。

### 9.2 BERT 查询分类（第二道智能闸门）

分类器使用微调后的 `bert-base-chinese`，输出两类：
- **通用知识**：不查询企业内部知识库，直接调用大模型回答（例如“写一个 Python 函数判断素数”）。
- **专业咨询**：进入 RAG 检索增强流程。

**设计价值**：
- 减少无效检索和噪声上下文。
- 低置信或边界模糊问题采用保守策略，优先进入 RAG。

### 9.3 多策略 RAG 检索（核心竞争力）

StrategySelector 根据问题特征动态选择策略：

| 策略           | 适用场景               | 实现方式                     |
|----------------|------------------------|------------------------------|
| 直接检索       | 大多数常规问题         | BGE-M3 混合检索 + Reranker   |
| 子查询检索     | 复合型问题             | LLM 生成多个子问题分别检索后合并 |
| 回溯问题检索   | 问题表述过于口语化     | LLM 生成更规范的回溯问题再检索 |
| HyDE           | 信息稀疏或高度抽象问题 | LLM 生成假设答案，用假设答案做检索 |

每种策略最终都走 `hybrid_search_with_rerank`（BGE-M3 召回 → Reranker 精排父文档）。

### 9.4 两阶段召回 + 上下文构建

1. BGE-M3 同时生成 dense 向量和 sparse 向量。
2. Milvus 执行 Hybrid Search + `source` 元数据过滤。
3. 命中子块后回到对应父块，保证上下文完整性。
4. Reranker 对候选父文档进行精排。
5. 取 Top-M 拼接成最终上下文 + 最近 5 轮历史 → 进入生成 Prompt。

### 9.5 生成约束与可靠性保障（已实现）

- Prompt 明确要求“优先基于检索上下文回答”。
- 检索不到明确依据时，模型倾向于返回“根据当前知识库未找到明确信息，建议结合现场情况人工确认”。
- 质量异常类问题引导模型以“可能原因 + 排查方向 + 建议确认”组织回答。
- 通过 RAGAS 进行离线 Faithfulness、Answer Relevance 等指标评估，作为持续调优依据。

**注意**：当前实现中，生成结果直接流式返回，未做服务端生成后一致性校验缓冲（这是与理想架构的主要差异，已在第 6.3 节说明）。

## 10. 业务与运行可靠性设计

### 10.1 已实现的可靠性措施
- 标准问答高置信阈值拦截。
- BERT 分类保守路由策略。
- 知识域 `source_filter` 强过滤。
- Reranker 二次筛选降低噪声。
- 最近 5 轮历史截断 + prompt 长度控制。
- Redis 缓存高频标准答案。

### 10.2 运行保障
- FastAPI + WebSocket 流式输出。
- MySQL 持久化会话历史。
- 请求超时、输入长度限制。
- `/health` 健康检查。
- 完整日志记录（分类结果、策略选择、检索命中数、耗时等）。

## 11. 评估指标

| 层次     | 核心指标                          | 说明 |
|----------|-----------------------------------|------|
| 标准问答 | FAQ 命中率、误命中率              | 高频问题是否稳定命中 |

| 检索层   | Hit@K、MRR、NDCG                  | 相关文档是否被召回并排前 |
| 生成层   | Faithfulness、Answer Relevance（RAGAS 离线） | 回答是否基于上下文 |
| 业务层   | 人工确认通过率、用户采纳意愿      | 员工是否认可答案 |
| 运行层   | 平均响应时间、超时率              | 内部使用是否顺畅 |

## 12. 演进规划（诚实版）

**已完成 / 当前可用**：
- 标准问答 + BERT 分类 + 多策略 RAG 完整链路打通

- 答案来源引用展示
