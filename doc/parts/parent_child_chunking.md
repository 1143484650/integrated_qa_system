# 父子块切分模块笔记

## 模块定位

父子块切分位于文档加载之后、向量入库之前：

```text
文档加载 / OCR
    ↓
父子块切分
    ↓
BGE-M3 向量化
    ↓
Milvus 入库
    ↓
子块检索 → 父块还原 → Reranker 精排 → LLM 生成
```

它解决的是 RAG 中一个常见矛盾：块太大，检索不精准；块太小，上下文不完整。

## 为什么需要父子块

如果直接用大块入库，优点是上下文完整，但用户问题可能只命中文档中的一小段，向量相似度会被大量无关内容稀释。

如果直接用小块入库，优点是检索命中更精准，但小块内容可能太碎，交给大模型生成答案时容易缺少前后文。

父子块的思路是：

- **父块负责上下文保真**：尽量保留段落、标题和上下文结构，适合提供给 Reranker 和大模型。
- **子块负责语义检索**：在父块内部按语义边界继续切成更小片段，更容易精准命中用户问题。
- **子块携带父块信息**：检索命中子块后，可以回到对应父块。

一句话概括：

> 子块提高召回精度，父块保证上下文完整。

## 常见切块策略

| 策略 | 做法 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 固定长度切分 | 按固定字符数或 token 数切块 | 实现简单，速度快 | 容易切断句子、段落和完整知识点 | 临时原型、格式很规整的文本 |
| 递归语义切分 | 按段落、换行、句子、标点等分隔符递归切分，长度超限再兜底 | 比固定长度更自然，能尽量保留语义边界 | 仍然偏规则，不是真正理解文本语义 | 普通中文文档、课程资料、技术文档 |
| 结构感知切分 | 按标题、章节、Markdown 层级、HTML 标签等结构切分 | 能保留文档层级和章节逻辑 | 依赖文档结构质量，非结构化文本效果有限 | Markdown、网页、规范文档、带标题的资料 |
| 父子块切分 | 先切大块作为父块，再在父块内切小块作为子块 | 兼顾检索精度和上下文完整性 | 存储会增加，链路更复杂 | RAG 知识库问答 |
| 模型切分 | 使用文本分割模型判断段落边界或语义边界 | 更接近语义理解，适合复杂文本 | 成本更高，速度更慢，部署更复杂 | 对切分质量要求很高的长文档 |

## 项目采用的切分策略

当前项目包装口径中采用的是：

> 父块结构保真 + 子块模型语义切分 + 长度兜底 + overlap 边界保留。

具体来说：

- 父块大小约 1200 字，按结构和段落尽量保留较完整上下文。
- 子块由 `nlp_bert_document-segmentation_chinese-base` 文档分割模型在父块内部做语义切分，尽量让每个子块对应一个完整的小知识点。
- 子块目标大小约 300 字，用来提高向量检索命中精度。
- 相邻块之间保留约 50 字 overlap，减少边界信息丢失。
- 最终入库检索的是子块，但每个子块都会携带父块内容。

这里的“语义切分”指的是使用 `nlp_bert_document-segmentation_chinese-base` 识别文本中的语义边界，例如段落主题变化、知识点边界、标题与正文关系等，再结合长度参数控制子块粒度。

## 语义切分话术细节

面试中说“子块用语义切分”时，可以表达成 **模型判断语义边界，长度参数做兜底**。

更准确的说法是：

> 子块不是简单按固定长度硬切，而是先通过文档分割模型识别语义边界，尽量把同一个知识点、同一段解释或同一个问答逻辑保留在一个子块里。如果某个语义片段仍然过长，再结合 `child_chunk_size` 做长度控制。这样可以让子块既足够小，适合向量检索，又尽量保持语义完整。

可以拆成三层解释：

1. **模型识别语义边界**：先判断一段文本内部哪些位置更像知识点边界、段落边界或主题切换点。
2. **长度作为兜底约束**：如果模型识别出的语义片段太长，再继续拆成更小片段，最终控制在约 300 字左右。
3. **overlap 保护边界信息**：相邻子块保留 50 字重叠，避免关键信息刚好落在切分边界时丢失。

如果面试官追问“为什么要用模型做子块切分”，可以回答：

> 因为子块是进入向量库检索的最小单位，如果只是固定长度切分，可能会把一个完整概念拆断，导致向量表示不稳定。模型语义切分可以尽量保证每个子块表达一个相对完整的小知识点，这样生成 embedding 时语义更集中，向量检索命中也更准确。

如果面试官追问“为什么不直接固定长度切”，可以回答：

> 固定长度切分实现简单，但容易把完整句子、代码解释或课程概念切断。我们用模型语义切分，可以优先保持知识点的完整性，再用长度参数控制 chunk 大小。这样切出来的子块更适合做向量检索，召回结果也更容易对应一个完整知识点。

如果面试官追问“父块和子块为什么用不同策略”，可以回答：

> 父块更关注上下文完整，所以主要保留文档结构和段落逻辑；子块更关注检索命中，所以用模型语义切分，把父块拆成更细、更聚焦的小知识点。这样父块负责给大模型完整上下文，子块负责让向量检索更精准。

## 关键参数

参数来自 `config.ini` 的 `[retrieval]` 配置：

| 参数 | 当前值 | 作用 |
|------|--------|------|
| `parent_chunk_size` | 1200 | 父块大小，保留结构和完整上下文 |
| `child_chunk_size` | 300 | 子块目标大小，模型语义切分后用于控制检索粒度 |
| `chunk_overlap` | 50 | 相邻块重叠，减少边界信息丢失 |
| `retrieval_k` | 3 | 检索阶段初步召回数量 |

对应配置：

```ini
[retrieval]
parent_chunk_size = 1200
child_chunk_size = 300
chunk_overlap = 50
retrieval_k = 3
```

## 元数据设计

每个子块会携带父块信息：

```python
sub_chunk.metadata["parent_id"] = parent_id
sub_chunk.metadata["parent_content"] = parent_doc.page_content
sub_chunk.metadata["id"] = f"{parent_id}_child_{k}"
```

关键字段含义：

| 字段 | 含义 |
|------|------|
| `id` | 子块 ID |
| `parent_id` | 父块 ID |
| `parent_content` | 父块完整内容 |
| `source` | 文档来源分类 |
| `file_path` | 原始文件路径 |
| `timestamp` | 加载时间 |

入库时，Milvus 中保存的是子块文本和父块元数据：

```python
"text": doc.page_content,
"dense_vector": embeddings["dense"][i],
"sparse_vector": sparse_vector,
"parent_id": doc.metadata["parent_id"],
"parent_content": doc.metadata["parent_content"],
```

## 检索阶段怎么使用

检索时先用子块做向量召回：

```text
用户问题
    ↓
BGE-M3 生成 dense / sparse 向量
    ↓
Milvus 检索子块
```

命中子块后，系统会根据 `parent_content` 还原父块：

```python
parent_docs = self._get_unique_parent_docs(sub_chunks)
```

还原逻辑是：

```python
parent_content = chunk.metadata.get("parent_content", chunk.page_content)
unique_docs.append(Document(page_content=parent_content, metadata=chunk.metadata))
```

然后对父块去重，再交给 BGE-Reranker 精排，最后取前几个父块作为大模型上下文。

完整链路可以概括为：

```text
子块向量检索
    ↓
父块内容还原
    ↓
父块去重
    ↓
BGE-Reranker 精排
    ↓
LLM 基于父块上下文生成答案
```

## 方案优势

- **检索更精准**：子块粒度小，更容易命中用户问题中的具体知识点。
- **上下文更完整**：最终给 Reranker 和 LLM 的是父块，不是孤立短句。
- **边界损失更少**：overlap 能缓解关键信息落在切分边界的问题。
- **适合混合检索**：子块可以同时生成 dense_vector 和 sparse_vector，适合 Milvus 混合召回。
- **和精排配合自然**：先用子块扩大召回，再用父块做 Reranker 精排，兼顾速度和准确率。

## 局限和取舍

- **存储会增加**：每个子块都保存了 `parent_content`，会带来额外存储成本。
- **参数需要调优**：1200 / 300 / 50 是当前项目包装中的推荐配置，不一定适合所有文档类型。
- **依赖前置解析质量**：OCR 或文档加载效果不好时，切分质量也会受影响。
- **模型切分有额外成本**：相比规则切分，语义分割模型需要额外推理开销，更适合在离线入库阶段完成。

## 面试说法

可以这样回答：

> 我们项目没有简单按固定长度把文档切碎，而是采用父子块分层切分。父块更偏结构保真，约 1200 字，用来保留段落、标题和完整上下文；子块则在父块内部通过文档分割模型做语义切分，目标大小约 300 字，让每个子块尽量对应一个完整的小知识点。相邻块有 50 字 overlap，减少边界信息丢失。最终入库检索的是子块，但子块会携带 `parent_id` 和 `parent_content`，检索命中后再回到父块给 Reranker 和大模型使用，所以能兼顾召回精度和回答完整性。

如果面试官问为什么子块要用模型语义切分，可以补充：

> 子块是向量检索的基本单位，它的语义越集中，embedding 表示越稳定。模型语义切分可以减少把一个知识点拆散的情况，让召回结果更容易对应用户问题。相比固定长度切分，这种方式更适合课程资料、技术文档和 FAQ 这类知识密集型文本。
