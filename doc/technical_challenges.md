## 难点总览

| 难点 | 当前做法 | 主要价值 | 主要风险 |
| --- | --- | --- | --- |
| 两级检索路由 | MySQL + BM25 先命中，失败再进 RAG | 标准问题稳定、快速返回 | 阈值判断偏经验，可能误命中或漏命中 |
| 混合检索 | BGE-M3 dense/sparse + Milvus | 同时兼顾语义和关键词 | 权重未系统评估 |
| 父子块 + Reranker | 子块召回，父块生成，再重排 | 兼顾命中精度和上下文完整性 | Reranker 延迟随候选数量上升 |
| 策略选择 | LLM 从四种检索策略中选一个 | 让不同问题走不同召回方式 | LLM 输出字符串不稳定 |
| 查询分类 | BERT 判断通用知识 / 专业咨询 | 减少不必要的 RAG 检索 | 误判会导致专业问题脱离知识库 |
| 流式输出 | WebSocket / SSE 逐 token 返回 | 改善长答案等待体验 | 异常信号和正常 token 混在一起 |

## 1. 两级检索路由

核心代码在 `new_main.py` 的 `IntegratedQASystem.query()`：

```python
answer, need_rag = self.bm25_search.search(query, threshold=0.85)
```

当前设计是先查 MySQL 标准问答库。BM25 高置信命中时直接返回标准答案；未命中时才调用 `RAGSystem.generate_answer()`。

这个设计适合教育问答场景：课程价格、学科介绍、常见 FAQ 这类问题需要稳定答案，不适合每次都交给大模型自由生成。

主要取舍：

- `threshold=0.85` 是经验阈值，倾向于“宁可走 RAG，也不要错答标准答案”。
- 一旦 BM25 命中，就不会再查询 RAG，即使知识库里可能有更新或更完整的内容。
- 当前只看 top1 是否超过阈值，没有比较 top1 和 top2 的差距。

后续优化：

- 增加 top1/top2 分差判断，分差太小时回退 RAG。
- 记录每次 BM25 命中的问题、分数和答案来源，便于后续调阈值。
- 对标准问答库增加更新流程，避免 MySQL 与 Redis 缓存内容不一致。

## 2. 混合检索与重排

RAG 检索核心在 `rag_qa/core/vector_store.py` 的 `VectorStore.hybrid_search_with_rerank()`。

当前检索链路是：

```text
BGE-M3 生成 dense/sparse 向量
  -> Milvus 两路检索
  -> RRF 排名融合
  -> 子块命中后还原父块
  -> BGE-Reranker 精排
  -> 取前 candidate_m 个父块进入 prompt
```

这个设计的价值在于：

- dense 向量适合语义相近但措辞不同的问题。
- sparse 向量适合课程名、技术名、缩写词等关键词。
- 子块检索更准，父块生成上下文更完整。
- Reranker 可以减少无关文档进入 prompt。

主要取舍：

- RRF 只使用排名，不直接使用 dense/sparse 原始分数；它更稳，但仍需要结合小评测集观察召回覆盖率。
- Reranker 是 CrossEncoder 思路，候选越多，延迟越高。
- 父块内容作为 metadata 跟随子块存储，会增加 Milvus 存储体积。

后续优化：

- 建一小批评测问题，对 RRF k 值、两路 topK 和进入 Reranker 的候选数量做验证。
- 记录 Milvus 检索耗时、Reranker 耗时和最终文档 ID。
- 当候选数量变大时，考虑轻量 Reranker、batch predict 或减少进入精排的候选数。

## 3. 策略选择链路

策略选择发生在 RAG 专业咨询分支中，相关代码是 `rag_qa/core/strategy_selector.py` 和 `rag_qa/core/new_rag_system.py`。

当前支持四种策略：

| 策略 | 作用 |
| --- | --- |
| 直接检索 | 原问题直接检索 |
| 假设问题检索 | 先生成假设答案，再检索 |
| 子查询检索 | 拆成多个子问题分别检索 |
| 回溯问题检索 | 把复杂问题简化成基础问题再检索 |

难点不在“有几种策略”，而在“策略选择是否稳定”。当前 `StrategySelector` 让 LLM 返回一个策略名称字符串，后续代码再根据字符串进入不同分支。

主要风险：

- LLM 可能返回不在枚举内的文本。
- LLM 可能带解释性内容，导致字符串不完全匹配。
- 非直接检索策略会额外触发 LLM 调用，增加端到端延迟。

后续优化：

- 增加策略白名单校验，不合法时显式降级到 `直接检索` 并记录 warning。
- 统计不同策略的触发次数、平均耗时、命中文档数量。
- 对简单问题优先使用规则判断，减少一次 LLM 策略选择调用。

## 4. 查询分类器

查询分类器在 `rag_qa/core/query_classifier.py`，作用是判断问题属于“通用知识”还是“专业咨询”。

它和策略选择器的职责不同：

- `QueryClassifier` 决定是否需要知识库。
- `StrategySelector` 决定专业咨询问题怎么检索。

当前价值是节省 RAG 成本。通用知识问题可以直接进入 LLM 生成，不必执行 embedding、Milvus 检索和 Reranker。

主要风险：

- 如果把专业咨询误判成通用知识，答案会脱离项目知识库。
- 二分类粒度较粗，像“AI 课程里讲哪些机器学习算法”这类问题容易混淆。
- 前置问候语规则和分类器有部分职责重叠。

后续优化：

- 增加“是否需要检索”的人工评测集。
- 对误判样本持续回流训练分类器。
- 在低置信分类结果下，优先进入 RAG，而不是直接相信通用知识分支。

## 5. 流式输出与异常传播

项目有两种流式入口：

- `app.py`：WebSocket `/api/stream`
- `api.py`：SSE `/query`

生成链路大致是：

```text
call_dashscope()
  -> RAGSystem.generate_answer()
  -> IntegratedQASystem.query()
  -> WebSocket / SSE 返回
```

流式输出解决的是用户体验问题：RAG 检索和 LLM 生成都需要等待，逐 token 返回可以让前端更快展示内容。

主要取舍：

- BM25 命中时是一次性答案，RAG 分支才是 token 级流式。
- `call_dashscope()` 发生异常时，错误信息会作为字符串继续向下游传递。
- 下游很难区分“正常 token”和“错误 token”。

后续优化：

- 统一 WebSocket 和 SSE 的事件格式。
- 给生成器增加结构化返回，例如 `{type: "token" | "error" | "end"}`。
- 对 LLM 超时、限流、认证失败等错误做明确错误码。

## 6. 缓存与会话管理

当前项目主要有三类状态数据：

| 类型 | 位置 | 作用 |
| --- | --- | --- |
| 标准问答数据 | MySQL `jpkb` | 保存高频标准问题和答案 |
| 缓存 | Redis | 缓存问题列表、分词结果、命中答案 |
| 会话历史 | MySQL `conversations` | 保存最近几轮问答历史 |

现有设计能支撑演示和教学场景，但还有几个工程风险：

- Redis 答案缓存按原始 query 字符串命中，同义问法无法复用。
- 问题列表和答案缓存缺少明确 TTL 或失效策略。
- `conversations` 只保存 question/answer，不保存检索文档、策略、耗时等调试信息。
- 每次写入历史后都会清理超过 5 轮的旧记录，高并发下需要关注表增长和删除成本。

后续优化：

- 给 Redis 缓存增加 TTL 或版本号。
- 引入语义缓存，用 query embedding 命中相似问题。
- 给会话记录增加 metadata 字段，保存策略名、检索文档 ID、耗时等信息。

## 7. 优先级建议

| 优先级 | 建议 | 原因 |
| --- | --- | --- |
| 高 | 给策略选择结果加白名单校验 | 成本低，能避免隐式降级 |
| 高 | 统一流式错误事件格式 | 否则错误会混入正常答案 |
| 中 | 记录检索和生成链路日志 | 后续调参需要真实数据 |
| 中 | 增加 BM25 top1/top2 分差判断 | 降低误命中标准答案的风险 |
| 中 | 建立小规模 RAG 评测集 | 判断权重、策略、rerank 是否真的有效 |
| 低 | 做语义缓存 | 有价值，但需要额外索引和淘汰策略 |
| 低 | 评估 Milvus 分布式和索引升级 | 当前更偏演示项目，暂不急 |

## 总结

这个项目的难点不是单个算法，而是多条链路之间的取舍：

- MySQL/BM25 保证标准问题稳定。
- RAG/Milvus 扩展开放问题覆盖面。
- BGE-M3、父子块和 Reranker 提升上下文质量。
- 查询分类和策略选择减少无效检索，但也引入误判和额外延迟。
- WebSocket/SSE 改善体验，但需要更清晰的错误信号。

后续最值得优先补的是观测能力：把每次请求走了哪条链路、命中了哪些文档、用了什么策略、耗时多少记录下来。没有这些数据，阈值、权重和策略选择都只能停留在经验判断。
