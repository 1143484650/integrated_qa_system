# 项目开发难点与解决方案

本文档用于面试时回答“项目开发过程中遇到过什么难点、怎么解决”的问题。回答时建议不要只说用了哪些技术，而是按“问题背景 -> 具体难点 -> 解决方案 -> 效果和权衡”的结构展开。

## 1. 多轮对话中历史上下文如何参与 RAG

### 问题背景

知识库问答不是所有问题都是完整单轮问题。用户经常会连续追问，例如：

```text
第 1 轮：覆膜起泡可能是什么原因？
第 2 轮：怎么排查？
```

第二轮问题本身信息很少，如果直接拿“怎么排查”去检索知识库，召回结果容易偏离原话题。

### 难点

最开始容易把“多轮对话”理解成把最近几轮历史直接拼进 prompt，但这只能帮助大模型生成时理解上下文，不能解决检索阶段的问题。

真正的难点是：

- 历史如果完全不参与检索，追问类问题召回不准。
- 历史如果每次都参与检索，又可能污染新问题。
- 历史过长还会增加 token 成本，并让模型注意力分散。

### 解决方案

我把历史上下文分成两种用途：

- **生成阶段默认使用历史**：每次加载最近 5 轮会话历史，拼入 prompt，帮助模型理解指代和追问。
- **检索阶段条件使用历史**：只有当前问题存在指代、省略或明显追问时，才结合历史进行问题改写。

例如：

```text
历史问题：覆膜起泡可能是什么原因？
当前问题：怎么排查？
改写后：覆膜起泡应该怎么排查？
```

改写后的完整问题再进入向量检索，可以明显提升召回准确性。

### 面试回答话术

> 我遇到的一个难点是多轮对话和 RAG 检索之间的关系。最开始我只是把最近几轮历史拼到 prompt 里，但发现这只能帮助生成，不能帮助检索。比如用户追问“怎么排查”，如果直接检索这个短问题，召回会很不稳定。后来我把历史分成生成阶段和检索阶段两种用途：生成阶段始终带最近 5 轮历史，检索阶段只有遇到指代、省略、追问时才触发问题改写，把短追问改写成完整问题后再检索。这样既支持连续追问，又避免新话题被旧历史污染。

## 2. 标准问答和 RAG 如何路由

### 问题背景

项目里既有 MySQL 标准问答库，也有文档型知识库 RAG。如果所有问题都走大模型，会增加成本和延迟；如果只走标准问答，又无法覆盖复杂问题。

### 难点

难点在于如何判断一个问题应该直接返回标准答案，还是进入 RAG 检索生成。

如果阈值太低，可能错误命中标准问答；如果阈值太高，又会让大量高频问题进入 RAG，降低效率。

### 解决方案

系统采用 MySQL + Redis + BM25 作为标准问答优先层：

- Redis 缓存问题列表、分词结果和高频答案。
- BM25 对用户问题和标准问题做相似度匹配。
- 只有相似度达到高置信阈值时，才直接返回标准答案。
- 没有可靠命中时，再回退到 RAG。

这样可以把高频、稳定、标准化的问题快速解决，把复杂问题交给 RAG。

### 面试回答话术

> 我没有让所有问题都直接走大模型，而是设计了标准问答优先的路由。高频稳定问题先走 MySQL 标准问答库，用 BM25 做相似问题匹配，Redis 缓存问题和答案。只有达到高置信阈值才直接返回，否则回退到 RAG。这个设计主要是为了降低延迟和成本，同时避免大模型对标准问题生成出不一致的答案。

## 3. 检索召回既要精确又要保留上下文

### 问题背景

RAG 的回答质量很大程度取决于检索质量。知识库文档通常比较长，如果切块太大，向量检索不够精确；如果切块太小，生成时又缺少上下文。

### 难点

核心矛盾是：

- 小块适合精确召回。
- 大块适合完整生成。
- 如果只用小块，答案容易断章取义。
- 如果只用大块，召回容易不准。

### 解决方案

项目采用父子块切分：

- 子块用于向量入库和精确命中。
- 子块命中后回溯到父块。
- 父块作为最终生成上下文。
- 再通过 reranker 对候选父块精排。

这样可以兼顾召回精度和生成上下文完整性。

### 面试回答话术

> 文档切块也是一个难点。刚开始如果切得太大，检索不精确；切得太小，模型生成时缺少上下文。我最后采用父子块策略：子块负责入库和召回，命中子块后回溯到父块，再用父块进入 prompt。这样检索阶段比较精确，生成阶段又能拿到完整上下文，减少断章取义。

## 4. 多策略检索的选择问题

### 问题背景

用户问题形态差异很大，有的问题关键词明确，有的问题抽象，有的问题是复合问题，还有的是多轮追问。

### 难点

如果所有问题都只走一种向量检索策略，效果不稳定：

- 明确问题适合直接检索。
- 抽象问题可能需要 HyDE。
- 复合问题适合拆成子查询。
- 追问问题适合回溯改写。

### 解决方案

系统引入策略选择器，根据问题特征选择不同检索方式：

- 直接检索：适合关键词清晰的问题。
- HyDE：适合抽象或上下文不足的问题。
- 子查询检索：适合多个关注点的问题。
- 回溯问题检索：适合多轮追问和指代不清的问题。

### 面试回答话术

> 我发现单一向量检索很难适配所有问题，所以做了多策略检索。问题明确时直接检索；问题比较抽象时用 HyDE 先生成假设答案再检索；复合问题拆成子查询；多轮追问则结合历史改写为完整问题。这样不同类型的问题可以走不同检索路径，比固定 pipeline 更稳定。

## 5. 生成答案的可靠性控制

### 问题背景

RAG 虽然能把知识库内容放进 prompt，但大模型仍然可能补充上下文里没有的内容，尤其是涉及工艺参数、排查步骤、业务结论时。

### 难点

难点不是让模型生成得更长，而是让模型只基于检索证据回答：

- 检索上下文不足时不能编造。
- 未被文档支持的内容需要过滤。
- 高风险问题应该提示人工确认。

### 解决方案

系统在生成后增加轻量 Faithfulness 校验思路：

- 先把大模型答案作为草稿。
- 根据检索上下文判断答案是否被证据支持。
- 对缺少依据的内容进行删除、拒答或转人工。
- 最终答案附带来源信息，方便追溯。

### 面试回答话术

> 生成可靠性是我比较重视的点。RAG 并不等于模型一定不会幻觉，所以我在设计上把 RAG 生成结果先看作草稿，再做答案和证据的一致性校验。对于上下文不支持的内容，不直接返回；如果依据不足，就返回“当前资料未明确说明，建议人工确认”。这样可以降低模型编造业务结论的风险。

## 6. 流式输出和生成后校验之间的冲突

### 问题背景

用户希望答案能流式输出，但可靠性设计又要求先完成生成后校验。

### 难点

如果模型一边生成一边直接返回给前端，后面即使发现答案有问题，也已经无法撤回。

### 解决方案

设计上把 RAG 答案分成两阶段：

- 服务端先缓冲草稿答案。
- 完成一致性校验、风险控制和引用组装。
- 最后再把通过校验的最终答案流式返回。

这会牺牲一部分首 token 延迟，但能换来更可靠的最终输出。

### 面试回答话术

> 这里有一个工程权衡：流式输出追求快，但生成后校验要求先拿到完整答案。如果边生成边发，后面发现有幻觉也撤不回来。所以我把 RAG 答案设计成服务端先缓冲草稿，完成 Faithfulness 校验和风险控制后，再把最终答案流式返回。这样牺牲一点首 token 延迟，但提高了可靠性。

## 7. 来源追溯和权限过滤

### 问题背景

企业知识库问答不能只返回答案，还要知道答案依据来自哪里，尤其是内部员工需要追溯文档来源。

### 难点

如果只返回自然语言答案，用户无法判断答案是否可信，也无法定位原始文档。

同时，不同用户可能只能访问部分知识域，不能把越权来源暴露出去。

### 解决方案

设计中为每个检索命中的 chunk 保留元数据：

- `source`
- `parent_id`
- `timestamp`
- `parent_content`

最终答案返回时组装 `citations` 和 `sources`，只展示本轮实际进入 prompt 且通过过滤的来源。

### 面试回答话术

> 企业内部知识库和普通聊天机器人不一样，答案需要可追溯。我在检索结果里保留 source、parent_id、timestamp 等元数据，最终答案会带 citations 和 sources。这样用户不仅能看到答案，还能知道依据来自哪个知识域和哪个父块。权限上也通过 source 过滤控制，避免返回用户无权访问的来源。

## 8. 总结回答模板

如果面试官只让简短回答，可以用下面这个版本：

> 我项目里最大的难点是 RAG 的工程稳定性，而不是单纯接一个大模型接口。比如多轮对话时，历史到底怎么参与检索和生成；文档切块时，怎么兼顾召回精度和上下文完整性；生成答案后，怎么避免模型编造没有依据的内容。我的解决方式是：用 MySQL + BM25 优先处理高频标准问答；RAG 检索采用父子块、混合检索和 reranker；多轮场景下只在追问时结合历史做问题改写；生成后增加答案-证据一致性校验和来源引用。这样系统不仅能回答问题，也更容易控制质量和追溯依据。
