# 策略选择与多轮会话设计

本文记录当前代码中“检索策略选择”和“多轮会话拼接”的实现方式，重点说明 history 在哪些阶段只是结构化传递，哪些阶段会真正拼成 prompt。

## 1. 总体链路

当前在线问答入口在 `IntegratedQASystem.query_result()`：

- 读取 `session_id` 对应的最近会话历史。
- 把 `query`、`source_filter`、`session_id`、`history` 传入 `QAGraphWorkflow`。
- 图编排完成后，把最终答案写回 MySQL 会话历史。

代码位置：

- `new_main.py:191`：`query_result()`
- `new_main.py:98`：`_fetch_recent_history()`
- `new_main.py:126`：`update_session_history()`

主流程：

```text
API / WebSocket
  -> query_result()
  -> get_session_history(session_id)
  -> qa_graph.run(query, source_filter, session_id, history)
  -> load_history
  -> standard_qa_search
  -> query_classification
  -> printing_keyword_review
  -> strategy_selector
  -> query_rewrite
  -> retrieve_docs
  -> generate_answer
  -> faithfulness_check
  -> build_citations / risk_control
  -> final_answer
  -> update_session_history()
```

## 2. 会话历史的存储设计

多轮会话通过 MySQL 表 `conversations` 保存。

代码位置：

- `new_main.py:52`：创建 `conversations` 表

表结构：

```sql
CREATE TABLE IF NOT EXISTS conversations(
    id INT AUTO_INCREMENT PRIMARY KEY,
    session_id VARCHAR(36) NOT NULL,
    question TEXT NOT NULL,
    answer TEXT NOT NULL,
    timestamp DATETIME NOT NULL,
    INDEX idx_session_id (session_id)
)
```

历史保留策略：

- 每个 `session_id` 保留最近 5 轮问答。
- 查询时按时间倒序取 5 条，再反转为正序。
- 每次新答案生成后写入历史，并删除超出 5 轮的旧记录。

这里的 history 是结构化列表，不是 prompt 字符串：

```python
[
    {"question": "覆膜起泡是什么原因？", "answer": "..."},
    {"question": "怎么排查？", "answer": "..."}
]
```

## 3. history 什么时候加载，什么时候拼接

history 在入口处加载，但不会立刻拼接。

### 3.1 入口只加载 history

`query_result()` 会先读取会话历史，然后传给 `qa_graph.run()`。

此时 history 仍然是 list：

```text
history: list[dict]
```

### 3.2 `load_history` 只截断，不拼接

代码位置：

- `rag_qa/core/qa_graph.py:256`：`load_history()`

`load_history` 节点只保留最近 5 轮：

```python
history = state.get("history") or []
return {"history": history[-5:]}
```

它不会把 history 拼成 prompt。

### 3.3 BM25 和 BERT 不使用 history

当前设计里，历史不会参与：

- BM25 标准问答匹配
- BERT 查询分类
- 普通直接检索

原因是历史容易污染当前问题，尤其是用户已经切换话题时。如果每次检索都拼历史，可能把新问题错误拉回旧话题。

## 4. 策略选择的实现方式

策略选择在 `QAGraphWorkflow.strategy_selector()` 中执行。

代码位置：

- `rag_qa/core/qa_graph.py:300`：`strategy_selector()`
- `rag_qa/core/strategy_selector.py:25`：`StrategySelector`
- `rag_qa/core/strategy_selector.py:55`：策略选择 prompt
- `rag_qa/core/strategy_selector.py:97`：`select_strategy()`

当前支持四种检索策略：

| 策略 | 使用场景 | 是否拼历史 |
| --- | --- | --- |
| 直接检索 | 问题明确，关键词清晰 | 否 |
| HyDE 假设答案检索 | 问题抽象、短、直接检索效果可能差 | 否 |
| 子查询检索 | 复合问题、多个关注点 | 否 |
| 回溯问题检索 | 多轮追问、指代不清、省略主题 | 是，进入 `query_rewrite` 后拼 |

策略选择分两层：

1. 如果当前问题像追问，并且有历史，直接选择“回溯问题检索”。
2. 否则调用 `StrategySelector.select_strategy(query)`，让 LLM 从四种策略中选一个。

追问判断代码位置：

- `rag_qa/core/qa_graph.py:496`：`_looks_like_follow_up()`

当前追问判断规则：

- 问题长度较短。
- 或包含“这个、那个、刚才、上面、前面、它、怎么排查、还有呢、为什么”等指代或追问词。

### 4.1 策略选择 Prompt

策略选择器本质上是一个意图路由 LLM，输入只有当前问题 `query`，输出只允许是一个策略名。

当前 prompt 的核心设计可以写成下面这种更具体的形式：

```text
你是一个检索策略选择器，负责根据用户问题，从以下四种检索策略中选择最合适的一种。

可选策略：
1. 直接检索
   - 适用于问题明确、关键词清晰、无需额外改写的场景。
2. 假设问题检索（HyDE）
   - 适用于问题较抽象、较短、直接检索效果可能不佳的场景。
3. 子查询检索
   - 适用于复合问题、多个实体、多个关注点需要分别检索的场景。
4. 回溯问题检索
   - 适用于多轮追问、指代不清、省略主题、依赖上一轮上下文的场景。

输出要求：
- 只能返回一个策略名称。
- 必须严格从上述四种策略中选择。
- 不要解释原因，不要输出分析过程，不要输出多余标点。

用户问题：
{query}
```

这个 prompt 的目的不是做问答，而是做路由判断，所以必须把：

- 候选策略范围固定住
- 每个策略的适用边界写清楚
- 输出格式限制死

### 4.1.1 这个 Prompt 的精髓

这个 prompt 的核心不是让 LLM 自由理解并回答，而是把它当成稳定的路由器来用。

具体有四点：

1. **任务收窄**
   - 只要求模型做一件事：从 4 个策略里选 1 个。
   - 不让它解释、不让它扩写、不让它顺手回答问题本身。

2. **候选集固定**
   - 先把“直接检索 / HyDE / 子查询 / 回溯问题检索”定义死。
   - 模型只能在这个集合里选，避免跑偏到其他模糊策略。

3. **边界写清楚**
   - 每个策略都给了适用场景和例子。
   - 尤其把“回溯问题检索”明确绑定到“多轮追问、指代不清、依赖上一轮上下文”。

4. **输出格式锁死**
   - 只返回策略名。
   - 不要分析过程，不要解释原因，不要输出多余文本。
   - 这样后面的图编排可以直接消费结果，不需要再做复杂解析。

一句话总结：

> 这个 prompt 的重点不是“答得好”，而是“选得稳、选得准、能被程序直接接住”。

### 4.2 安全与稳定性设计

策略选择这里的安全重点，不是内容安全，而是“路由稳定性”。

当前主要有三层控制：

1. **低温度调用**
   - `strategy_selector.py` 里调用 DashScope 时使用 `temperature=0.1`。
   - 目的，是减少同一个问题在多次调用时出现不同策略的概率。

2. **规则优先**
   - 如果 `history` 不为空，并且问题明显像追问，图编排会直接选择“回溯问题检索”。
   - 这一步不依赖 LLM，目的是让多轮追问更稳定，避免模型把短追问误判成普通直答。

3. **输出约束**
   - prompt 要求只能返回策略名称。
   - 当前代码在调用异常时会默认退回到“直接检索”，避免策略选择失败导致整条链路中断。

### 4.3 当前实现与可增强点

当前实现已经能覆盖主流程，但还可以再加一层白名单校验：

```python
VALID_STRATEGIES = {
    "直接检索",
    "假设问题检索",
    "子查询检索",
    "回溯问题检索",
}
```

如果模型返回了非法文本，可以强制降级成“直接检索”。这一步目前还没写进代码，属于后续加固项。

## 5. 问题重写和历史拼接

问题重写不是独立策略，而是“回溯问题检索”策略中的执行步骤。

代码位置：

- `rag_qa/core/qa_graph.py:309`：`query_rewrite()`
- `rag_qa/core/qa_graph.py:500`：`_format_history()`

只有当策略是“回溯问题检索”且存在 history 时，才会拼接历史会话：

```text
历史对话：
Q: 覆膜起泡是什么原因？
A: 覆膜起泡通常与胶水干燥不足、覆膜温度控制不当有关。

当前问题：
怎么排查？

只输出改写后的问题，不要解释。
```

期望得到：

```text
覆膜起泡应该怎么排查？
```

然后 `retrieve_docs()` 使用 `rewritten_query` 检索。

代码位置：

- `rag_qa/core/qa_graph.py:329`：`retrieve_docs()`

如果策略不是“回溯问题检索”，检索阶段不会拼接历史。

## 6. LLM 生成阶段的历史拼接

生成阶段每次都会拼接最近历史。

代码位置：

- `rag_qa/core/qa_graph.py:352`：`generate_answer()`
- `rag_qa/core/qa_graph.py:500`：`_format_history()`

生成 prompt 使用三类信息：

```text
检索上下文 context
当前问题 query
最近 5 轮历史 history
```

这里拼接 history 的目的不是检索，而是让 LLM 理解连续对话语境，例如：

- “还有呢？”
- “为什么？”
- “那怎么处理？”
- “刚才说的第二点展开讲讲。”

## 7. 检索阶段和生成阶段的区别

多轮会话在两个地方发挥作用，但目的不同。

| 阶段 | 是否拼 history | 目的 |
| --- | --- | --- |
| 入口加载 | 否 | 从 MySQL 取最近历史，放入 `GraphState.history` |
| BM25 | 否 | 避免标准问答被历史污染 |
| BERT 分类 | 否 | 只判断当前问题类型 |
| 策略选择 | 不拼完整历史 | 只判断是否像追问 |
| 问题重写 | 条件拼接 | 把追问改写成完整独立问题 |
| RAG 检索 | 使用改写后的问题 | 提升追问召回准确性 |
| LLM 生成 | 始终拼接 | 让回答理解上下文 |
| 历史保存 | 否 | 保存最终问答结果 |

## 8. 示例

### 第 1 轮

```text
Q: 覆膜起泡是什么原因？
```

处理方式：

```text
BM25 未命中
-> BERT 判定为专业问题
-> strategy_selector 选择直接检索
-> retrieve_docs 使用原问题检索
-> generate_answer 拼接空 history + context + question
-> 保存问答历史
```

### 第 2 轮

```text
Q: 怎么排查？
```

处理方式：

```text
入口读取上一轮 history
-> BM25 不使用 history
-> BERT 不使用 history
-> strategy_selector 发现问题短且有 history
-> 选择“回溯问题检索”
-> query_rewrite 拼接 history + 当前问题
-> 改写为“覆膜起泡应该怎么排查？”
-> retrieve_docs 使用 rewritten_query 检索
-> generate_answer 拼接 history + context + 原始问题
-> 保存最终答案
```

## 9. 当前边界

当前实现已经支持：

- `session_id` 级多轮会话。
- 最近 5 轮历史。
- 追问场景的问题重写。
- 生成阶段带历史。
- 回答后写回 MySQL。

当前还没有实现：

- 历史相关性筛选。
- 话题切换检测。
- 同一 `session_id` 的并发请求排队或加锁。
- BM25 FAQ 结合历史理解追问。
- 更细粒度的多轮评估指标。

当前设计原则是：

> history 先以结构化 list 形式进入 `GraphState`，只有“回溯问题检索的问题重写”和“最终 LLM 生成”两个 LLM 节点才会拼成 prompt。检索阶段默认不拼历史，只有追问场景才通过改写后的 `rewritten_query` 参与检索。
